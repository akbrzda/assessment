## 1. Обзор системы

### Тип системы

Корпоративный движок сертификации и обучения (LMS/Certification Engine). Backend — Node.js/Express, БД — MySQL (mysql2/promise + connection pool). Архитектура: Routes → Controller → Service/Model → Database.

### Ключевые сущности

| Сущность     | Таблица(ы)                                             | Описание                          |
| ------------ | ------------------------------------------------------ | --------------------------------- |
| User         | `users`                                                | Сотрудник / менеджер / суперадмин |
| Assessment   | `assessments`                                          | Аттестация с временными рамками   |
| Question     | `assessment_questions` + `assessment_question_options` | Вопросы и варианты ответов        |
| Attempt      | `assessment_attempts`                                  | Попытка прохождения               |
| Answer       | `assessment_answers`                                   | Ответы в рамках попытки           |
| Assignment   | `assessment_user/branch/position_assignments`          | Таблицы назначения                |
| Course       | `courses`, `course_topics`, `course_assessments`       | Иерархия курсов                   |
| Gamification | `user_stats`, `user_badges`, `gamification_rules`      | Очки, уровни, бейджи              |

### Уровень сложности

**Высокий.** Многоуровневая модель назначений (пользователь / должность / филиал), движок правил геймификации, таймер на серверной стороне, версионирование теорий.

## 2. Аудит модели данных

### ✅ Сильные стороны

- **Транзакционные операции с явным rollback** — создание/обновление аттестаций, создание попыток, сохранение ответов оборачиваются в `beginTransaction() / commit() / rollback()`. Целостность на уровне операций обеспечена.
- **Параметризованные запросы везде** — все запросы используют placeholders (`?`), SQL-инъекции исключены на уровне кода.
- **Миграции в отдельных файлах** — 10 миграций с нумерацией и осмысленными именами. Историю схемы можно восстановить.
- **Мягкое удаление попыток через статус** — попытки не удаляются физически, а переводятся в `cancelled`. История сохраняется.
- **Versioning теории** — `assessment_theory_versions` с флагом `completion_required` и привязкой конкретной версии к аттестации через `current_theory_version_id`.
- **`is_direct` в `assessment_user_assignments`** — различение прямого назначения от автоматического (через филиал/должность) позволяет безопасно пересчитывать назначения без потери явных записей.

### ❌ Проблемы

- **Снепшот вопросов для попытки отсутствует.** При старте попытки в `assessment_attempts` не сохраняется ни порядок, ни содержимое вопросов. Вопросы всегда загружаются напрямую из `assessment_questions`. Если администратор отредактирует аттестацию во время активной попытки (через `updateAssessment`, который делает `DELETE FROM assessment_questions WHERE assessment_id = ?` и вставляет заново), порядок и содержимое вопросов для пользователя изменятся на середине теста. Это нарушает честность и воспроизводимость аттестации.
- **Порядок вопросов для возобновлённой попытки не фиксируется.** Рандомизация при старте не реализована (`logWithTime("createAttempt", "Создана попытка без рандомизации")`), но задокументированное требование предполагает её добавление. Когда рандомизацию добавят, без снепшота порядок изменится при перезаходе.
- **Дублирование логики оценки ответов.** Функции `saveAnswer` и `saveAnswersBatch` содержат полностью идентичный код оценки каждого типа вопроса (единственный, множественный, текст, сопоставление) — суммарно ~200 строк дублированного кода. Любое исправление бага в одном месте требует ручного повтора в другом. Именно это стало причиной существующего бага в `matching`.
- **Переменная `committed` объявляется, но не используется** в `updateAssessment` — в функции нет блока `if (!committed)` для rollback в `finally`. При сетевой ошибке после `commit()` и до `release()` поведение непредсказуемо (хотя в данном случае транзакция уже завершена, это мёртвый код, вводящий в заблуждение).
- **`deleteAssessment` — одна строка без проверок.** `pool.execute("DELETE FROM assessments WHERE id = ?", [assessmentId])`. Нет проверки наличия активных попыток перед удалением. Если FK не настроены каскадно, удаление оставит осиротевшие записи в `assessment_attempts`, `assessment_answers`, `assessment_questions`.

### ⚠️ Критические риски

- **Нет снепшота назначений в `assessment_user_assignments` на момент начала попытки.** При изменении должности или перемещении сотрудника в другой филиал `assignUserToMatchingAssessments` пересоздаёт все `is_direct = 0` записи. Пользователь, у которого уже есть активная попытка, может потерять или получить доступ к аттестациям без предупреждения.
- **`selected_option_ids` хранится как JSON-строка в VARCHAR/TEXT** — нет возможности делать запросы по этому полю, нет типовой валидации. При анализе ошибок в ответах потребуется парсинг в приложении.

### 💡 Рекомендации

- Добавить таблицу `assessment_attempt_questions` для фиксации снепшота вопросов и их порядка в момент старта попытки. Добавить индекс по `(attempt_id, order_index)`.
- Вынести логику оценки одного ответа в единую функцию `scoreAnswer(question, payload)` и вызывать её из `saveAnswer` и `saveAnswersBatch` — единый источник истины.
- В `deleteAssessment` добавить проверку на `hasAttempts(assessmentId)` и возвращать 409, если попытки есть (метод уже существует в модели).

## 3. Аудит бизнес-логики

### Анализ: логика назначения и прогресс

- Назначение аттестации реализовано через три типа: прямое (пользователь), по должности, по филиалу. Пересечение (филиал + должность) обрабатывается отдельно. Логика корректна.
- `assignUserToMatchingAssessments` вызывается при создании/обновлении пользователя — автоматически привязывает к активным аттестациям. Транзакционно.

#### ❌ Логические пробелы

- **Редактирование активной аттестации не заблокировано.** В `update` есть комментарий и переменная `const canEditParameters = true;` — явный признак незаконченной логики. Текущая реализация позволяет изменить `pass_score_percent`, `time_limit_minutes`, `max_attempts` для аттестации с активными попытками. Это нарушает правила игры на середине теста: у пользователя меняется проходной балл или лимит времени.
- **Статус аттестации вычисляется динамически из `UTC_TIMESTAMP()`** в каждом запросе через `CASE WHEN`. Нет персистентного поля `status` в таблице. При изменении временной зоны сервера MySQL или при переводе часов все аттестации мгновенно меняют статус без уведомлений.
- **`cancelExpiredAttempts` вызывается при каждом GET-запросе** (`listForUser`, `getForUser`, `getDetail`, `startAttempt`). Это write-операция на каждый read-запрос. При 100 одновременных пользователях, каждую секунду просматривающих список аттестаций, база получает 100 лишних UPDATE-операций в секунду.
- **Нет серверного автозавершения по таймеру.** Истечение времени обнаруживается только при следующем обращении пользователя (через `cancelExpiredAttempts`). Если пользователь закрыл браузер за 10 секунд до истечения времени и не вернулся — попытка зависает в `in_progress` до следующего GET-запроса к этой аттестации. Попытка не засчитывается как использованная до момента обнаружения.
- **Параллельный старт попытки — race condition.** В `createAttempt`:
  1. Проверяется наличие `in_progress` попытки (без `FOR UPDATE` на `assessment_attempts`)
  2. Блокируется строка `assessments` через `FOR UPDATE`
  3. Вставляется новая попытка

  Два параллельных запроса могут одновременно пройти шаг 1 (оба не найдут активной попытки), затем оба заблокируются на шаге 2 (последовательно), и оба создадут по попытке. Пользователь получит `attemptNumber = N` и `N+1`, обе в статусе `in_progress`. Лимит попыток будет нарушен.

#### ⚠️ Сценарии сбоев

- **Обновление аттестации с удалением вопросов при активных попытках**: администратор редактирует аттестацию → `DELETE FROM assessment_questions WHERE assessment_id = ?` → пользователь загружает следующий вопрос → получает пустой список вопросов. Попытка зависает.
- **Пользователь переводится в другой филиал во время активной попытки**: `assignUserToMatchingAssessments` удалит `is_direct = 0` назначения → доступ к аттестации из старого филиала исчезнет → при следующем запросе `getAssessmentForUser` вернёт 404 → попытка не может быть завершена.

#### 💡 Исправления

- Запретить изменение `pass_score_percent`, `timeLimitMinutes`, `max_attempts` для аттестаций со статусом `active` или при наличии `in_progress` попыток.
- Вынести `cancelExpiredAttempts` из request-обработчиков в фоновый cron-job (раз в минуту через `setInterval` или node-cron). Это снизит write-нагрузку и гарантирует завершение даже без входящих запросов.
- В `createAttempt` добавить `FOR UPDATE` на запрос существующих `in_progress` попыток: `SELECT id FROM assessment_attempts WHERE assessment_id = ? AND user_id = ? AND status = 'in_progress' FOR UPDATE`. Это гарантирует атомарность проверки.

## 4. Движок аттестации и сертификации

### ❌ Проблемы

- **Критический баг в оценке `matching`-вопросов.** В `saveAnswer` и `saveAnswersBatch` ветвление типов выглядит так:

  ```
  if (question_type === "single") { ... }
  else if (question_type === "multiple" || question_type === "matching") { ... }  // ← matching попадает сюда
  else if (question_type === "text") { ... }
  else if (question_type === "matching") { ... }  // ← мёртвый код, никогда не выполняется
  ```

  Вопросы типа `matching` обрабатываются веткой `multiple || matching` с логикой множественного выбора: сравниваются `optionIds` как Set. Отдельная ветка с обработкой `matchPairs` является мёртвым кодом. В результате все ответы на вопросы-сопоставления оцениваются неверно.

- **Текстовые ответы — точное совпадение.** `normalizeTextAnswer` нормализует пробелы и приводит к нижнему регистру, но проверяет строгое равенство. Ошибка в одну букву, синоним, другой порядок слов — всё засчитывается как неправильный ответ. Требование из `features.md` — «слова-триггеры или примерные ответы» — не реализовано.

- **`max_attempts = 0` — безлимитный режим** (комментарий в коде: `max_attempts = 0 означает безлимитный режим`). Однако в `baseSchema` Joi-валидация требует `min(1)` и `max(3)`. Если `isUnlimitedAttempts` во фронтенде устанавливает `maxAttempts = 0`, Joi вернёт ошибку валидации 422. Механизм безлимитных попыток существует в модели, но API-слой не позволяет его передать.

- **Результат попытки содержит `correctAnswers` и `totalQuestions`, но не возвращает детали по вопросам** через `completeAttempt`. API клиенту отдаёт только цифры; пользователь не видит, какие вопросы ошибочны — только общий счёт. Детальный разбор (`getAttemptResult`) существует как отдельный эндпойнт, но вероятно не вызывается из экрана результатов.

#### ⚠️ Риски эксплуатации

- **Пользователь может отправить `completeAttempt` без ответов на все вопросы.** Нет проверки на количество данных ответов относительно количества вопросов. Можно начать попытку, сразу вызвать `/complete`, получить `0%` — это использует одну попытку, но при достаточном числе попыток можно злоупотреблять для изучения вопросов без риска.
- **Геймификация на уровне ответов (`processAnswerEvent`) обрабатывается в основном потоке запроса**, хотя и обёрнута в `try/catch`. Медленный движок правил задержит ответ клиенту. При 20 вопросах и активных правилах геймификации — 20 последовательных вызовов `rulesEngine.evaluate` на один тест.

#### 💡 Рекомендации

- Исправить ветвление в `saveAnswer`: убрать `matching` из условия `multiple || matching` и оставить только отдельную ветку с обработкой `matchPairs`. Исправление идентично для `saveAnswersBatch`.
- Ввести мягкую логику для текстовых ответов: список синонимов/ключевых слов в `correct_text_answer` через разделитель (`|`) и проверку включения хотя бы одного.
- Исправить Joi-схему: `maxAttempts: Joi.number().integer().min(0)` (0 = безлимит) с документированием поведения.
- Вынести обработку `processAnswerEvent` в фоновую очередь (или хотя бы `setImmediate`) — это не блокирующая операция для пользователя.

## 5. Роли и права доступа

### ❌ Проблемы

- **Middleware `canEditUser` не проверяет принадлежность пользователя филиалу менеджера** (зафиксировано в TASKS.md п. 2.1). Любой менеджер через PUT `/users/:id` может редактировать пользователей из чужих филиалов, если знает их ID.
- **Нет unit-теста или integration-теста на изоляцию менеджера по филиалам** — уязвимость существует, но не покрыта тестами, поэтому может быть не замечена при ревью.
- **Маршрут `/invitations` в роутере admin-frontend** ограничен только `superadmin`, хотя по требованиям менеджер должен создавать инвайты для своего филиала. Несоответствие между frontend-роутером и задокументированными требованиями.
- **`verifyAdminRole` и `requireRole` — два разных middleware** для одного и того же. Риск несогласованного применения: один разработчик использует одно, другой — другое, уровень доступа будет отличаться.
- **`checkModuleAccess` — матрица модулей не централизована.** Согласно TASKS.md п. 2.3, значения по умолчанию для `role=manager` расходятся между backend и admin-frontend. Если добавить новый модуль, нужно обновить два места — вероятность рассинхронизации высока.

### ⚠️ Риски безопасности

- **JWT-секреты с fallback-значениями.** В `backend/src/config/env.js` (по TASKS.md) есть `|| 'your_secret_key_here'`. Если переменные не заданы в production, сервер запускается с предсказуемыми секретами — любой может подделать JWT-токен с полными правами суперадмина.
- **CORS c `origin: true` при пустом `ALLOWED_ORIGINS`** — принимает запросы с любого домена. Критично для API с cookie-auth (refresh token).
- **`SameSite` у refresh-cookie и CSRF:** без double-submit cookie или проверки `Origin` на `POST /admin/auth/refresh` любой сайт может выполнить CSRF-атаку на обновление токена.
- **Telegram `auth_date` не проверяется** — initData старше нескольких часов принимается как валидный. Перехваченный initData можно использовать повторно.
- **Access token в `localStorage`** (TASKS.md п. 1.5) — XSS в admin-frontend позволяет украсть токен с полными правами администратора.

### 💡 Исправления

- В `canEditUser`: добавить `WHERE u.id = ? AND u.branch_id = manager.branch_id`. Реализовать единожды в middleware, а не в каждом контроллере.
- Добавить fail-fast при старте: `if (!process.env.JWT_SECRET) { console.error('JWT_SECRET not set'); process.exit(1); }`.
- Объединить `verifyAdminRole` и `requireRole` в один middleware с параметром роли.
- Вынести матрицу модулей в общий конфигурационный файл `roleModules.js` и импортировать в оба места.

## 6. Аудит API-дизайна

### ❌ Плохие практики

- **`console.log` в production-коде.** Сотни вызовов `console.log(...)` и `logWithTime(...)` в `assessmentModel.js` (в каждой функции: `[createAttempt]`, `[saveAnswer]`, `[listAssessmentsForManager]` и т.д.). В production это:
  - замедляет I/O (синхронная запись в stdout)
  - засоряет логи чувствительными данными (userId, attemptId, тело запроса)
  - усложняет поиск реальных ошибок
- **`submitAnswer` возвращает `204 No Content`** — клиент не получает информации о корректности ответа в реальном времени. Это сознательное решение для предотвращения «угадывания», но тогда API-контракт нигде явно не задокументирован.
- **Эндпойнт `POST /assessment/:id/attempts/:attemptId/answers/batch`** дублирует `POST /assessment/:id/attempts/:attemptId/answers`. Два эндпойнта, одна логика, риск рассинхронизации поведения. Уже воспроизведён в баге с matching.
- **Статус аттестации не возвращается как enum со строгими значениями.** Код использует строки `'pending'`, `'active'`, `'closed'` — они не задокументированы как контракт. Смена имени в SQL-`CASE` сломает клиент без предупреждения.
- **Ошибки без `code` — только `error: "текст"`.** Кроме одного исключения (`THEORY_NOT_COMPLETED`), все ошибки возвращают только строку. Клиент не может программно различить «аттестация закрыта» от «превышен лимит попыток» — только парсит текст на русском языке.
- **`GET /assessments/:id` в admin вызывает `cancelExpiredAttempts`** — GET-запрос имеет побочный эффект (write). Нарушает принцип idempotency GET.

### 💡 Улучшения

- Заменить `console.log` на структурированный логгер (`winston` или `pino`) с уровнями: `debug` только в development, `error/warn` в production.
- Добавить machine-readable `code` к каждой ошибке: `{ error: "...", code: "ATTEMPT_LIMIT_EXCEEDED" }`.
- Убрать `cancelExpiredAttempts` из GET-обработчиков, перенести в cron.
- Оставить один эндпойнт для сохранения ответов (batch), убрать дублирующий single-answer endpoint или сделать его оберткой над batch.

## 7. Масштабируемость и производительность

### ⚠️ Узкие места

- **`findAssessmentByIdForManager` выполняет 7 последовательных SQL-запросов на один GET-запрос:**
  1. Основной SELECT аттестации
  2. Вопросы и варианты ответов
  3. Должности
  4. Филиалы
  5. Пользователи (прямые назначения)
  6. Участники с лучшим результатом (сложный `ROW_NUMBER()` subquery)
  7. (неявно через `hasAttempts`)

  При 50 одновременных менеджерах, открывающих детали аттестации, это 350 запросов к БД. Без кеширования это станет первым bottleneck.

- **`listAssessmentsForUser` содержит 4 коррелированных `UNION`-подзапроса** с 8 копиями `userId`. Запрос выполняется при каждом открытии списка аттестаций. При 500 сотрудниках с разными должностями/филиалами этот запрос станет медленным из-за отсутствия составных индексов.

- **Назначение через `assignUserToMatchingAssessments` — N запросов к БД:**
  - До 3 SELECT-запросов для определения пересечения
  - 1 DELETE + 1 batch INSERT
  - Вызывается при каждом создании/обновлении пользователя
  - При массовом обновлении (смена должности всему филиалу) — N вызовов последовательно.

- **Геймификация на каждый ответ — синхронный запрос в основном потоке.** `processAnswerEvent` открывает соединение с БД, начинает транзакцию, вызывает `rulesEngine.evaluate`. При 20 вопросах в аттестации — 20 последовательных DB-транзакций только для геймификации.

- **Отсутствие пагинации в `listAssessmentsForManager` и `listAssignableUsers`.** При 1000+ пользователей и 200+ аттестациях запросы загружают все строки в память Node.js-процесса.

### 💡 Рекомендации

- Добавить составные индексы: `(assessment_id, user_id)` на `assessment_attempts`, `(user_id)` на `assessment_user_assignments`, `(branch_id)` на `users`.
- Внедрить кеширование результата `findAssessmentByIdForManager` на 30–60 секунд (Redis или in-memory LRU). Данные аттестации меняются редко.
- Добавить серверную пагинацию: `LIMIT ? OFFSET ?` + заголовок `X-Total-Count`.
- Геймификацию перевести в фоновую очередь (bull/bullmq + Redis) — события накапливаются и обрабатываются асинхронно.
- Для `cancelExpiredAttempts`: cron раз в 60 секунд вместо вызова при каждом запросе.

## 8. Рассогласование Backend и UX

Следующие проблемы бэкенда напрямую создают сбои в пользовательском опыте:

| Проблема бэкенда                                    | Последствие для UX                                                                                                                                                                             |
| --------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `matching` оценивается как `multiple`               | Пользователь правильно сопоставил все пары → видит «неправильно». Доверие к системе разрушено                                                                                                  |
| `cancelExpiredAttempts` только при GET              | Пользователь закрыл браузер за 30 сек до конца → попытка висит `in_progress`. При следующем входе статус непоследователен — «продолжить» нельзя (таймер вышел), завершить нельзя автоматически |
| Нет снепшота вопросов                               | При возобновлении попытки вопросы могут быть в другом порядке или исчезнуть (если аттестацию отредактировали). Пользователь видит неожиданное поведение                                        |
| Редактирование активной аттестации не заблокировано | Менеджер меняет `pass_score_percent` на 90% для аттестации, которую сотрудник уже проходит с расчётом на 70%. Результат оценивается по новым правилам                                          |
| `submitAnswer` возвращает 204                       | Клиент не знает, сохранился ли ответ. При плохом соединении пользователь видит «тишину» и не понимает, принят ли ответ                                                                         |
| Пагинация отсутствует                               | При 500+ пользователях GET `/users` может timeout'иться — менеджер видит пустой экран или ошибку загрузки                                                                                      |
| JWT в localStorage                                  | Уязвимость к XSS — потенциальная компрометация admin-сессии без видимых симптомов для пользователя                                                                                             |

## 10. Топ-5 критических исправлений

### 🔴 P0 — Немедленно (данные некорректны / безопасность нарушена)

**1. Исправить баг оценки `matching`-вопросов**

В `saveAnswer` и `saveAnswersBatch` убрать `"matching"` из ветки `multiple || matching`:

```js
// Было:
else if (question.question_type === "multiple" || question.question_type === "matching") { ... }

// Стать должно:
else if (question.question_type === "multiple") { ... }
else if (question.question_type === "matching") { /* matchPairs-логика */ }
```

Требуется идентичное исправление в обоих местах. После исправления вынести в общую функцию `scoreAnswer`.

**2. Закрыть уязвимости JWT и CORS**

- Добавить fail-fast при старте: если `JWT_SECRET` / `JWT_REFRESH_SECRET` не заданы — `process.exit(1)`
- Убрать `origin: true` при пустом `ALLOWED_ORIGINS`, выбросить ошибку
- Добавить проверку `auth_date` в `telegramAuthService`: окно 5 минут

### 🟠 P1 — В течение текущего спринта (нарушает честность аттестации)

**3. Устранить race condition при параллельном старте попытки**

В `createAttempt` перед `FOR UPDATE` на `assessments` добавить:

```sql
SELECT id FROM assessment_attempts
WHERE assessment_id = ? AND user_id = ? AND status = 'in_progress'
FOR UPDATE
```

Это гарантирует, что два параллельных запроса не создадут две одновременных попытки.

**4. Перенести `cancelExpiredAttempts` в cron-job**

Убрать вызов из всех request-обработчиков. Добавить:

```js
setInterval(() => assessmentModel.cancelExpiredAttempts(), 60_000);
```

при старте сервера. Дополнительно: реализовать `completeExpiredAttempts` — завершать с подсчётом результата вместо простой отмены.

### 🟡 P2 — Следующий спринт (производительность и целостность)

**5. Добавить снепшот вопросов для попытки**

Создать таблицу `assessment_attempt_questions (attempt_id, question_id, order_index)` и заполнять при старте. При возобновлении загружать вопросы через JOIN с этой таблицей, а не напрямую из `assessment_questions`. Это фиксирует порядок и защищает от изменения аттестации на середине попытки.
