## 1. Резюме состояния продукта

Платформа аттестаций (Панда Пицца LMS) находится на стадии **рабочего MVP с накопленным техническим долгом**. Ключевые функциональные блоки реализованы: аттестации с таймером, геймификация, роли, курсы, лидерборд. Однако система содержит:

- **Критические баги**, напрямую разрушающие достоверность результатов аттестаций
- **Незакрытые уязвимости безопасности**, недопустимые в production
- **Гонки состояний** в движке попыток, позволяющие обходить лимиты
- **UX-тупики** в ключевых сценариях для обеих ролей (admin и employee)
- **Отсутствие пагинации**, делающее систему неработоспособной при реальной нагрузке

**Общая оценка системы:** 4–5 / 10 по надёжности и UX-зрелости.

**Стратегическое направление:** Переход от MVP к продуктово зрелой платформе через три этапа:
1. Стабилизация (исправление критических багов и безопасности)
2. Укрепление фундамента (архитектурные и UX-улучшения)
3. Масштабирование (производительность, аналитика, продуктовые возможности)

## 2. Карта проблем

### A. Баги движка аттестации (нарушают достоверность данных)

| Проблема | Почему важно | Что будет, если не исправить |
|---|---|---|
| Matching-вопросы оцениваются как multiple-choice | Все аттестации с matching выставляют ошибочные баллы | Решения об аттестации сотрудников принимаются на основе неверных данных |
| Нет снепшота вопросов при старте попытки | При возобновлении или редактировании аттестации порядок/содержание вопросов меняется | Нарушение честности аттестации, дезориентация пользователя |
| Автозавершение по таймеру только при следующем GET | Попытка зависает в `in_progress` при закрытии браузера | Неопределённый результат, засорение статистики |
| Race condition при параллельном старте попытки | Два параллельных запроса создают две активные попытки | Обход лимита попыток, некорректный подсчёт |
| Редактирование активной аттестации не заблокировано | Менеджер меняет проходной балл на середине теста | Сотрудник оценивается по правилам, о которых не знал |

### B. Безопасность и права доступа

| Проблема | Почему важно | Что будет, если не исправить |
|---|---|---|
| JWT-секреты с fallback-значениями | При незаданных env-переменных сервер запускается с публичными секретами | Любой может подделать JWT-токен суперадмина |
| CORS с `origin: true` при пустом ALLOWED_ORIGINS | API принимает запросы с любого домена | Уязвимость к CSRF-атакам через куки refresh-токена |
| `canEditUser` не проверяет `branch_id` менеджера | Менеджер редактирует пользователей из чужих филиалов | Нарушение конфиденциальности, GDPR-риск |
| Telegram `auth_date` не проверяется | Перехваченный initData принимается повторно без ограничения по времени | Replay-атака на авторизацию через Telegram |
| Access token в `localStorage` | XSS-атака компрометирует admin-сессию | Полный захват прав суперадмина |

### C. Архитектура и производительность

| Проблема | Почему важно | Что будет, если не исправить |
|---|---|---|
| `cancelExpiredAttempts` вызывается на каждый GET | 100 одновременных пользователей = 100 write-операций/сек при просмотре списков | Деградация производительности БД под нагрузкой |
| 7 последовательных SQL-запросов на детали аттестации | При 50 менеджерах = 350 запросов к БД одновременно | Первый bottleneck при реальной нагрузке |
| Нет серверной пагинации ни в одном списке | 500+ пользователей — весь список загружается в память процесса | Таймауты, зависание интерфейса, OOM |
| Синхронная геймификация на каждый ответ | 20 вопросов = 20 DB-транзакций в основном потоке запроса | Заметное замедление ответа API при прохождении теста |
| `selected_option_ids` как JSON-строка в VARCHAR | Нельзя делать запросы по этому полю | Сложность аналитики и диагностики ответов |

### D. UX и информационная архитектура (Администратор)

| Проблема | Почему важно | Что будет, если не исправить |
|---|---|---|
| Инвайты доступны только superadmin | Менеджер не может делегировать вход своим сотрудникам | Дополнительная ручная нагрузка на суперадмина |
| Числа без контекста («Завершено: 12» без denominator) | Менеджер не видит охват — 12 из 12 или 12 из 400? | Управленческие решения на основе неполных данных |
| Нет пагинации в UI (списки пользователей, вопросов) | При 500+ записях интерфейс деградирует | Практическая неработоспособность для крупных филиалов |
| Два режима геймификации без явного индикатора | Непонятно, какая система активна | Хаос в настройках, риск двойного начисления |
| Нет массовых действий над пользователями | Управление 100+ сотрудниками — только поодиночке | Значительные временные затраты на рутину |
| Нет глобального поиска | Найти пользователя или вопрос — только через конкретный раздел | Снижение эффективности работы администратора |
| Форма создания аттестации — один длинный scroll | Добавление вопросов внизу страницы при 20+ вопросах неудобно | Ошибки при создании, потеря данных |
| Нет хлебных крошек на глубоких страницах | Навигация «назад» через кнопку браузера нарушает IA | Дезориентация, лишние клики |

### E. UX и информационная архитектура (Сотрудник)

| Проблема | Почему важно | Что будет, если не исправить |
|---|---|---|
| Дашборд показывает только одну ближайшую аттестацию | При 3 открытых аттестациях остальные скрыты | Сотрудник пропускает дедлайны |
| Точки навигации по вопросам не кликабельные | Нельзя вернуться к вопросу для исправления ответа | Снижение качества прохождения, лишняя тревожность |
| Нет правильных ответов на экране результатов | Сотрудник не понимает ошибки | Повторная попытка не ведёт к обучению |
| Matching-вопросы через `<select>` на мобильном | Telegram Mini App — мобильный интерфейс, нативный select неудобен | Высокое трение при прохождении ключевого типа вопросов |
| Нет истории попыток в профиле | Нет обзора своих результатов по времени | Слабая вовлечённость, ощущение «чёрного ящика» |
| Маршрут `/registration` доступен при запрете на самостоятельную регистрацию | Потенциальная лазейка для создания несанкционированных аккаунтов | Нарушение контроля доступа |

### F. Дизайн-система и консистентность

| Проблема | Почему важно | Что будет, если не исправить |
|---|---|---|
| Два источника стилей кнопок (`<Button>` + `.btn`) | При обновлении дизайна нужно менять в двух местах | Визуальное рассогласование, рост технического долга CSS |
| Нет rich empty states с CTA | Пустые экраны ведут в тупик | Новые пользователи не понимают, что делать дальше |
| Нет глобального toast для серверных ошибок | Разные компоненты по-разному показывают ошибки | Непредсказуемый опыт при ошибках API |
| Ошибки API без machine-readable `code` | Клиент парсит текст на русском языке для программных решений | Хрупкость фронтенда, сложность интернационализации |

### G. Операционные и процессные пробелы

| Проблема | Почему важно | Что будет, если не исправить |
|---|---|---|
| `console.log` в production-коде повсеместно | Засоряет логи чувствительными данными, замедляет I/O | Невозможность диагностики реальных проблем в prod |
| Нет unit-тестов на изоляцию менеджера по филиалам | Уязвимость `canEditUser` существует, но не обнаруживается ревью | Уязвимость будет воспроизведена в новых контроллерах |
| Два middleware для одного (`verifyAdminRole` + `requireRole`) | Разные разработчики применяют разные, доступ расходится | Несогласованный контроль доступа |
| Матрица модулей не централизована | Backend и frontend хранят дефолты для manager отдельно | Рассинхронизация при добавлении нового модуля |
| Дублирование логики оценки ответов (saveAnswer / saveAnswersBatch) | Баг matching уже воспроизведён именно из-за этого | Каждый следующий баг нужно исправлять в двух местах |

P0 назначен задачам, где существующая проблема либо искажает данные (matching баг, race condition), либо создаёт активную уязвимость безопасности. Без закрытия P0 любая аналитика по аттестациям недостоверна, а безопасность системы не может считаться обеспеченной.

P3 — стратегические улучшения и оптимизации.

## 7. Быстрые победы vs глубокие изменения

### Быстрые победы (LOW усилие, HIGH влияние)

| Задача | Почему быстро | Почему важно |
|---|---|---|
| Баг matching: исправление ветвления | ~30 мин работы, два файла | Исправляет достоверность всех аттестаций с matching |
| Fail-fast для JWT-секретов | ~10 строк кода | Закрывает критическую уязвимость безопасности |
| CORS исправление | ~5 строк конфигурации | Закрывает CSRF-риск |
| Проверка `auth_date` | ~5 строк | Устраняет replay-атаку Telegram |
| `cancelExpiredAttempts` → cron | Перемещение вызова, + `setInterval` | Снижает write-нагрузку в 100x |
| Удалить маршрут /registration | 1 строка в роутере | Закрывает лазейку |
| Счётчик активных аттестаций на дашборде | Небольшое изменение компонента | Устраняет пропуск дедлайнов |
| Числа N/M в таблице аттестаций | +1 поле в API + изменение шаблона | Менеджер мгновенно видит охват |
| Хлебные крошки | Переиспользуемый компонент | Навигация без потери контекста |
| Исправить Joi-схему max_attempts | 1 строка | Разблокирует безлимитный режим |

### Среднесрочные улучшения (MEDIUM усилие, STRUCTURAL значение)

| Задача | Почему среднесрочно | Почему важно |
|---|---|---|
| Снепшот вопросов попытки | Миграция + изменение нескольких методов | Основа честности всей системы аттестации |
| Пагинация (API + UI) | Затрагивает несколько эндпойнтов и компонентов | Без этого система неработоспособна при росте |
| Перенос access token из localStorage | Синхронизация frontend + backend | Убирает XSS-вектор для admin-сессий |
| AssessmentWizard как default | Переработка роутинга + UI | Устраняет главную точку трения при создании аттестаций |
| Унификация middleware | Рефакторинг роутов | Предотвращает будущие уязвимости доступа |
| Rich empty states | Дизайн + реализация для каждого модуля | Устраняет навигационные тупики системно |
| Глобальный toast (100%) | Аудит + замена во всех компонентах | Консистентный опыт при ошибках |

### Глубокие изменения (HIGH усилие, СТРАТЕГИЧЕСКОЕ значение)

| Задача | Почему глубоко | Почему важно |

## 8. Зависимости и порядок выполнения

```
[Фаза 0: Диагностика] → параллельно → [Фаза 1]

Фаза 1 (независимые блоки, выполняются параллельно):
  [1.1 Matching bug fix] ─────────────────────────────────────┐
  [1.2 JWT + CORS + Telegram auth_date] ─────────────────────┤
  [1.3 canEditUser branch isolation] ────────────────────────┤
  [1.4 Race condition FOR UPDATE] ───────────────────────────┤─→ [Фаза 2]
  [1.5 cancelExpiredAttempts → cron] ─────────────────────────┤
  [1.6 Удалить /registration] ────────────────────────────────┘

Фаза 2 (часть задач параллельно):
  [2.1 Миграция attempt_questions] → [2.2 Снепшот при старте] → ...
  [2.3 Блокировка редактирования активных аттестаций] (независимо)
  [2.4 Пагинация API] → [2.5 Пагинация UI] (последовательно)
  [2.6 Объединение middleware] → [2.7 Матрица модулей] → [2.8 Инвайты manager]
  [2.9 Access token из localStorage] (независимо, синхронизация FE+BE)

Фаза 3 (большинство независимы, после Фазы 2):
  [3.1 Правильные ответы на результатах] — ЗАВИСИТ от [1.1 Matching fix]
  [3.2 AssessmentWizard] — независимо
  [3.3 Rich empty states] — независимо, нужны макеты от дизайна
  [3.4 Числа N/M] — ЗАВИСИТ от обновления API-ответа аттестаций
  [3.5 Machine-readable codes] — лучше до активной разработки на фронте
  [3.6 История попыток] — ЗАВИСИТ от [2.4 Пагинация]

Фаза 4 (после Фазы 3):
  [4.1 Redis queue] — ЗАВИСИТ от DevOps: Redis в инфраструктуре
  [4.2 Кеширование] — ЗАВИСИТ от стабилизации схемы данных
  [4.3 Массовые действия] — ЗАВИСИТ от [2.4 Пагинация UI]
```

**Что нельзя начинать без продуктового решения:**
- Пересчёт matching-попыток (влияет на статус сотрудников)
- Онбординг (требует сценария от PM)
- Правильные ответы на экране результатов (политика: всегда показывать или только при провале?)

**Что можно делать параллельно:**
- Все задачи Блока 1.2 (безопасность) независимы друг от друга
- UX-улучшения admin и learner можно вести одновременно
- Пагинация API и дизайн empty states — независимые потоки

## 9. Команда и ответственность

### Product Manager
**Владеет:** Приоритезацией, продуктовыми решениями, коммуникацией с бизнесом  
**Проверяет:** Все задачи, затрагивающие правила аттестации (пересчёт matching, блокировка редактирования, показ правильных ответов)  
**Должен решить до начала разработки:** Политика пересчёта исторических данных; что происходит при удалении пользователя; когда показывать правильные ответы — всегда или только после провала

### Team Lead / Tech Lead
**Владеет:** Архитектурными решениями, ревью кода, разбивкой задач на спринты  
**Проверяет:** Снепшот попыток, race condition fix, объединение middleware, стратегию кеширования  
**Должен решить:** Выбор между LRU in-memory и Redis для кеша; стратегия миграции для существующих попыток без снепшота

### Backend Developer
**Владеет:** Всеми задачами backend — scoring, безопасность, пагинация, миграции, cron, кеш  
**Проверяет:** Все изменения API-контракта (совместно с FE)  
**Коллаборирует:** С Frontend при изменении структуры ответов API; с DevOps при env и миграциях

### Frontend Developer (Admin)
**Владеет:** Admin-панель — списки, формы, роутинг, wizard, права  
**Проверяет:** Соответствие UI продуктовым требованиям (с PM), соответствие макетам (с Designer)  
**Коллаборирует:** С Backend при пагинации и новых полях; с Designer для empty states и wizard

### Frontend Developer (Learner)
**Владеет:** Telegram Mini App — прохождение аттестаций, профиль, лидерборд  
**Проверяет:** UX сценарий аттестации от начала до результата (с QA)  
**Коллаборирует:** С Backend при экране результатов и истории попыток; с Designer для matching-UX

### UX/UI Designer
**Владеет:** Макеты для новых и изменённых сценариев  
**Проверяет:** Все задачи категории UX до начала разработки  
**Должен подготовить до Фазы 3:** Экран результатов с разбором; rich empty states; wizard creation flow; matching drag-and-drop (Фаза 4)

### QA Engineer
**Владеет:** Тест-планами, regression-suite, E2E сценариями  
**Проверяет:** Каждый P0/P1 блок перед выходом в production  
**Коллаборирует:** С Backend для тестирования race condition и scoring; с Frontend для E2E прохождения аттестации

### DevOps
**Владеет:** Production env-конфигурацией, применением миграций, инфраструктурой  
**Проверяет:** Все задачи с меткой «env», «миграции», «Redis»  
**Должен сделать до Фазы 1:** Проверить и заполнить JWT_SECRET, JWT_REFRESH_SECRET, ALLOWED_ORIGINS в production

## 10. Реестр рисков

| Риск | Причина возникновения | Влияние | Митигация |
|---|---|---|---|
| Пересчёт matching-попыток меняет аттестационный статус сотрудников | Bug существовал с момента запуска | Высокое (HR / юридические последствия) | Согласовать с бизнесом до пересчёта; зафиксировать дату применения исправления; уведомить HR |
| Исправление CORS ломает prod при незаполненном ALLOWED_ORIGINS | DevOps не успел настроить env до деплоя кода | Критичное (сервис недоступен) | Строгая последовательность: сначала env, потом деплой; чеклист перед деплоем |
| Миграция attempt_questions нарушает работу активных попыток | Попытки в `in_progress` не имеют снепшота | Высокое (ошибки при прохождении) | Fallback-логика: если снепшот отсутствует — загружать из `assessment_questions` (старый путь) |
| Race condition fix не полностью устраняет проблему | `FOR UPDATE` требует корректной настройки isolation level в MySQL | Среднее | Нагрузочное тестирование параллельных запросов после исправления |
| Перенос токена из localStorage вызывает логаут пользователей | При смене стратегии хранения текущие сессии инвалидируются | Низкое-среднее (UX) | Запланировать в период низкой нагрузки; уведомить администраторов |
| Пагинация API ломает существующие клиенты | Frontend ожидает полный список; при добавлении `X-Total-Count` поведение меняется | Среднее | Добавить пагинацию как opt-in (`?page=` без параметра = старое поведение) на переходный период |
| Неполное покрытие типов вопросов тестами | QA пропустил граничный случай при проверке scoring | Среднее | Добавить unit-тесты `scoreAnswer` по всем типам вопросов до деплоя Фазы 1 |
| Дублирование правил геймификации (легаси + движок) | Два параллельных механизма начисления очков | Среднее (финансовые/мотивационные последствия) | В рамках Фазы 3 провести аудит: какой механизм фактически активен; явно деактивировать второй |
| Рост технического долга CSS при параллельной разработке | Два разработчика добавляют стили в двух местах одновременно | Низкое-среднее | Договорённость в команде: только компонент `<Button>` до завершения унификации (Фаза 3) |
| Недооценка сложности фоновой очереди геймификации | Redis + bull в инфраструктуре требует DevOps-работы | Среднее (задержка Фазы 4) | Начать с `setImmediate` как промежуточным шагом; Redis-очередь — второй итерацией |

## 12. Финальные рекомендации

### Начать немедленно

1. **Исправить баг matching-вопросов.** Это не технический долг — это ошибка, которая прямо сейчас искажает результаты аттестаций и влияет на решения о персонале. Каждый день промедления — это день некорректных данных в production.

2. **Закрыть JWT fallback и CORS.** Эти уязвимости существуют с момента запуска. При незаданных env-переменных любой может подделать токен суперадмина. Исправление занимает менее часа — откладывать нельзя.

3. **Перенести `cancelExpiredAttempts` в cron.** Write-операция при каждом GET-запросе — это архитектурная проблема, которая замедляет систему прямо сейчас. Исправление — одна строка при старте сервера.

### Запланировать в ближайший спринт

4. **Снепшот вопросов попытки.** Без этого любая будущая рандомизация вопросов сломает возобновление попыток. Чем дольше откладывается, тем больше попыток в базе без снепшота — сложнее миграция.

5. **Серверную пагинацию.** Система работает только потому, что данных пока мало. При реальном росте (сотни сотрудников, сотни вопросов) первый же тяжёлый GET положит API. Пагинацию нужно строить до, а не после роста.

6. **Изоляцию менеджера по branch_id.** Это уязвимость, зафиксированная в TASKS.md. Значит, команда о ней знает. Любой инцидент с данными чужого филиала — это репутационный риск.

### Планировать системно (Фазы 3–4)

7. **UX-улучшения learner и admin.** Они не блокируют работу системы, но формируют доверие к продукту. Правильные ответы на экране результатов, кликабельные точки, счётчик аттестаций — это небольшие изменения с высоким воспринимаемым качеством.

8. **Структурированный логгер.** Без этого диагностика production-проблем — это поиск иголки в стоге сена из `console.log`. Особенно критично при росте нагрузки.

### Никогда не игнорировать

9. **Race condition при параллельном старте попытки.** Сейчас незаметен — мало пользователей. При нагрузке или стресс-тестировании создаст дубли попыток, нарушая честность аттестаций. Требует `FOR UPDATE` — несложное исправление с высоким долгосрочным эффектом.

10. **Дублирование логики scoring.** Именно из-за него matching-баг существует в двух местах. Даже после исправления баг вернётся при любом новом типе вопроса, если не вынести `scoreAnswer` в общую функцию. Это не рефакторинг ради рефакторинга — это устранение структурной причины проблемы.
