# 1. EXECUTIVE SUMMARY

## Что представляет проект

Модуль «Курсы» — система последовательного обучения сотрудников внутри LMS-платформы. Пользователь проходит курс по иерархии: **Курс → Тема курса → Подтема → Тест темы → Итоговая аттестация**. Администратор создаёт курсы через wizard, управляет публикацией и отслеживает аналитику.

## Основная цель

Запустить полнофункциональный модуль обучения с корректной логикой прогресса, блокировок, черновиков и аналитики — единым связным продуктом поверх существующей инфраструктуры аттестаций.

## Ожидаемая ценность

- Сотрудники проходят структурированное обучение в рамках платформы без внешних инструментов
- Управляющие управляют содержанием курсов и видят прогресс команды
- Все процессы (назначения, попытки, статусы) согласованы с существующим модулем аттестаций

## Подход к реализации

Инкрементальная поставка по фазам с жёстким следованием существующим архитектурным паттернам (Vue 3, MySQL, Express). Переиспользование логики аттестаций для тестов.

## Оценка сложности

**Высокая.** Проект затрагивает все три приложения (backend, admin-frontend, frontend), содержит нетривиальную бизнес-логику пересчёта прогресса и статуса `completed_locked`.

## 5 критических факторов успеха

1. **Терминология** — единое использование «Тема курса / Подтема» без «модуль/раздел» во всём коде и UI
2. **Пересчёт прогресса** — корректная обработка вставки новых обязательных подтем в уже пройденные темы
3. **Черновик изменений** — изоляция правок от опубликованной версии курса до явной публикации
4. **Переиспользование аттестаций** — тесты тем используют существующий flow, не изобретают новый
5. **API-контракты** — фиксация контрактов до старта frontend-разработки, без переделок на середине спринта

## Нефункциональные требования

- **Производительность**: список курсов ≤ 500ms; пересчёт прогресса асинхронный, не блокирует UI
- **Безопасность**: RBAC — пользователь видит только назначенные курсы; admin endpoint защищён ролью
- **Надёжность**: автосохранение черновика — retry при ошибке; потеря данных недопустима
- **Масштабируемость**: пересчёт прогресса при публикации изменений — батчевый, не in-request
- **Mobile-first**: все пользовательские экраны адаптированы для мобильных устройств
- **Доступность**: focus-state, контрастность, кликабельные зоны ≥ 44px

# 4. ПОТОКИ РАБОТ (WORKSTREAMS)

**Назначение**: API для всех операций с курсами, прогрессом, черновиками, аналитикой  
**Ключевые поставки**:

- Механизм черновика изменений (`course_drafts`)
- API таймера чтения подтем
- Пересчёт `completed_locked` при публикации изменений
- API лога изменений
- Аналитические эндпоинты (воронка, детальный отчёт)

**Зависимости**: схема БД (миграции) должна быть готова до frontend-интеграции

## WS-2: Admin Frontend

**Назначение**: Инструмент управляющего/администратора для создания и аналитики курсов  
**Ключевые поставки**:

- Доработка wizard (шаг 4 preview, drag-and-drop, обложки, теги)
- Черновик изменений в UI
- Страницы аналитики (список, курс, пользователи)
- Лог изменений

**Зависимости**: backend API черновика; компонент DragDrop

## WS-3: User Frontend

**Назначение**: Пользовательский интерфейс прохождения курса  
**Ключевые поставки**:

- Вкладки «Курсы / Аттестации» на странице обучения
- Список курсов (карточки)
- Страница курса (обновлённая)
- Страница темы курса
- Страница подтемы с таймером
- Тест темы (переиспользование аттестации)
- Status pages, BottomSheet

**Зависимости**: backend API подтем, таймера; компоненты BottomSheet, StatusBadge

## WS-4: QA

**Назначение**: Тестирование всех сценариев, edge-cases, регрессия  
**Ключевые поставки**:

- Тест-кейсы на все статусы и блокировки
- Регрессия mobile
- Проверка edge-cases (вставка подтемы, исчерпание попыток, срок действия)

## WS-5: Database / Migrations

**Назначение**: Схема для новых сущностей и изменений  
**Ключевые поставки**:

- Таблица `course_drafts`
- Поля для обложек, тегов, категорий курсов

- Таблица `course_change_log`

## ЭПИК E1: Пользовательский маршрут прохождения курса

**Бизнес-цель**: Сотрудник проходит курс от начала до итоговой аттестации в единном последовательном flow  
**Приоритет**: ВЫСОКИЙ  
**Зависимости**: WS-1 (API подтем, таймера), WS-5 (миграции)  
**Риски**: Сложность логики `completed_locked`; реактивность таймера на мобильных  
**Definition of Done**: Пользователь проходит полный маршрут от «Не начат» до «Завершён» без ошибок; все edge-cases обработаны
