## 1. ССТЬ — ТС С (P0/P1)

### 1.1 JWT-секреты — fail-fast

- [ ] брать fallback-значения (`your_secret_key_here`) из `backend/src/config/env.js`
- [ ] обавить проверку при старте: если `JWT_SECRET` или `JWT_REFRESH_SECRET` не заданы — аварийно завершать процесс с понятной ошибкой
- [ ] брать fallback из `backend/src/middleware/verifyJWT.js` и `backend/src/controllers/adminAuthController.js`

### 1.2 CORS — строгий origin для production

- [ ] апретить `origin: true` при пустом `ALLOWED_ORIGINS` (`backend/src/app.js`)
- [ ] обавить валидацию: если `ALLOWED_ORIGINS` не задан в production — аварийно завершать процесс
- [ ] бедиться, что WebSocket также получает строгий `allowedOrigins`

### 1.3 CSRF-защита для refresh cookie-flow

- [ ] еализовать double-submit cookie или проверку `Origin`/`Referer` на маршруте `POST /admin/auth/refresh`
- [ ] бедиться, что `SameSite` у refresh-cookie корректно выставлен для production

### 1.4 Rate-limit на admin login

- [ ] одключить `express-rate-limit` на `POST /admin/auth/login`
- [ ] астроить: не более 10 попыток за 15 минут с одного IP, затем блокировка на 15 минут
- [ ] обавить информативный ответ при блокировке (429 + сообщение)

### 1.5 Access token — перевод из localStorage в память

- [ ]  `admin-frontend/src/stores/auth.js` хранить access token в реактивной переменной (не в `localStorage`)
- [ ]  `admin-frontend/src/utils/axios.js` брать токен из store, а не из `localStorage`
- [ ] бедиться, что после refresh токен обновляется в памяти корректно

### 1.6 Telegram initData — проверка auth_date

- [ ]  `backend/src/services/telegramAuthService.js` добавить проверку `auth_date`
- [ ] кно: если `auth_date` старше 5 минут относительно текущего времени — отклонять запрос (401)

### 1.7 Security headers (helmet + CSP)

- [ ] одключить пакет `helmet` в `backend/src/app.js`
- [ ] астроить базовый CSP, запрещающий `unsafe-inline` для скриптов
- [ ] роверить, что `X-Frame-Options`, `HSTS`, `X-Content-Type-Options` выставлены
