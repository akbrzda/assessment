# UX/UI Аудит — Платформа аттестаций и обучения

> Дата: апрель 2026  
> Аудитор: AI-агент (Senior UX Auditor)  
> Версия системы: текущая (по состоянию на момент аудита)

---

## 1. Обзор продукта

### Тип системы

Корпоративная LMS-платформа для управления аттестациями, обучением и геймификацией сотрудников (идентифицирована как система для сети Панда Пицца). Реализована в виде двух интерфейсов:

- **Admin-панель** — веб-интерфейс для superadmin и manager
- **Мини-приложение (фронтенд)** — Telegram Mini App для сотрудников (employee)

### Роли пользователей

| Роль         | Интерфейс         | Доступ                               |
| ------------ | ----------------- | ------------------------------------ |
| `superadmin` | Admin-панель      | Полный доступ ко всем модулям        |
| `manager`    | Admin-панель      | Ограниченный доступ (свой филиал)    |
| `employee`   | Telegram Mini App | Прохождение аттестаций, геймификация |

### Сложность

**Высокая.** Система включает: аттестации с таймером, теорию, банк вопросов, курсы, геймификацию (уровни, бейджи, очки), лидерборд, инвайт-систему, отчётность с графиками, ролевой доступ.

## 2. UX-аудит администраторской части

### ✅ Сильные стороны

- **Двойной режим отображения (таблица / карточки)** — списки аттестаций и пользователей адаптированы под мобильные устройства с помощью двух отдельных шаблонов. Это правильный подход для корпоративного инструмента, который будет использоваться и с телефонов.
- **Активные теги фильтров в модуле пользователей** — отображение применённых фильтров в виде удаляемых тегов (`filter-tag`) снижает когнитивную нагрузку и делает состояние фильтрации явным.
- **Inline-статистика аттестаций в списке** — колонки «Назначено», «Завершено», «Средний балл» прямо в таблице позволяют менеджеру оценить статус без входа в детали.
- **Сайдбар с коллапсом** — возможность свернуть сайдбар до иконок освобождает рабочую зону, что важно при большом количестве данных на экране.
- **Защита удаления аттестации** — кнопка «Удалить» доступна только для статуса `pending`, что предотвращает случайное уничтожение активных или завершённых данных.
- **Оффлайн-баннер на дашборде** — система информирует об отсутствии сети и показывает кэшированные данные.

### ❌ Проблемы

- **Дашборд не показывает тренды.** «Топ-3 сотрудников по очкам» и «Средний процент успешности» отображаются без временного контекста. Непонятно: это рейтинг за неделю, месяц или за всё время. Метрика теряет ценность без точки отсчёта.
- **Нет индикатора причины провала аттестации.** В списке аттестаций отображается «Средний балл», но нет быстрого способа понять, какой процент сотрудников провалился из-за таймаута, а какой — из-за неправильных ответов. Поддержка не может диагностировать проблемы без входа в детали каждого пользователя.
- **Числа без контекста в таблице аттестаций.** Колонка «Завершено» показывает абсолютное число (например, «12»), без понимания, сколько это от назначенных. Менеджер не может мгновенно оценить охват — нужен формат «12 / 40» или прогресс-бар.
- **Отсутствие пагинации в явном виде.** В модулях Users, Assessments, Questions не видна серверная пагинация или бесконечная прокрутка. При 500+ пользователях или вопросах интерфейс, вероятно, зависнет или загрузит все данные разом.
- **Устаревшая и новая геймификация существуют одновременно.** На вкладке «Общие настройки» расположены поля «Начисление очков» с пометкой «(легаси)», а на вкладке «Геймификация» — новый движок правил. Пользователь не понимает, какая система активна в данный момент и не конфликтуют ли они.
- **Нет массовых действий.** В модуле пользователей нельзя назначить роль, удалить или экспортировать нескольких пользователей сразу. При управлении 100+ сотрудниками это создаёт значительную ручную нагрузку на менеджера.
- **Инвайты доступны только superadmin.** Согласно требованиям, менеджер должен создавать инвайты для своего филиала, но в роутере маршрут `/invitations` имеет `roles: ["superadmin"]`. Это блокирует делегирование.
- **Форма создания аттестации — длинный single-page form.** Все секции (основная инфо, назначение, вопросы) находятся на одной длинной странице без прогресс-индикатора. При создании аттестации с 20+ вопросами добавление вопросов находится в самом низу, прокрутка неудобна. Нет возможности перетаскивать вопросы для изменения порядка.
- **Вопросы в банке вопросов не связаны визуально с аттестациями.** Из списка вопросов непонятно, в какие аттестации включён вопрос. При редактировании вопроса возможно непреднамеренное изменение уже запущенных аттестаций.
- **Отсутствует глобальный поиск.** Найти пользователя, аттестацию или вопрос можно только войдя в соответствующий раздел. При растущей базе данных это снижает эффективность работы администратора.

### ⚠️ Критические риски

- **Менеджер потенциально видит данные других филиалов.** В коде аудита (TASKS.md) зафиксировано: middleware `canEditUser` не проверяет `branch_id`. Менеджер теоретически может просматривать и редактировать пользователей из чужих филиалов через прямые запросы к API. Это нарушение конфиденциальности данных.
- **Удаление пользователя необратимо.** В таблице пользователей кнопка «Удалить» (корзина) не имеет явного предупреждения о последствиях: что произойдёт с историей попыток, очками, инвайтами пользователя? Потеря исторических данных о сотрудниках — критический риск для HR-аудита.
- **Матч-вопросы оцениваются некорректно.** Баг в логике `matching` (зафиксирован в TASKS.md) означает, что аттестации с вопросами на сопоставление выставляют неверные баллы. Это делает результаты аттестаций ненадёжными.

### 💡 Рекомендации

- Добавить временной контекст к метрикам дашборда: «за последние 7 дней / 30 дней», с возможностью переключения. Показывать дельту (↑ 12% к прошлому периоду) для метрик активности.
- В таблице аттестаций заменить «Завершено: N» на «Завершено: N / M (XX%)» — сразу видно охват.
- Ввести предупреждение при удалении пользователя с указанием: «История попыток и баллы будут сохранены/удалены» — и требовать ввода подтверждающего текста для необратимых действий.
- Разделить настройки геймификации на два состояния с явным индикатором: «Режим: классический / Режим: движок правил» — и скрыть неактивный раздел.
- Переработать форму создания аттестации в многошаговый Wizard (уже есть компонент `AssessmentWizard.vue` — его нужно продвигать как основной путь).
- Реализовать серверную пагинацию с явным отображением: «Показано 25 из 312».

## 3. UX-аудит пользовательского интерфейса (Learner)

### ✅ Сильные стороны

- **Прогресс-полоска по вопросам** — во время аттестации верхняя полоска и счётчик «Вопрос X из N» устраняют неопределённость о продолжительности теста.
- **Точечная навигация (Question Dots)** — в нижней части экрана видны все вопросы с индикацией «отвечено / текущий», что позволяет отслеживать прогресс визуально.
- **Таймер с визуальным предупреждением** — при остатке < 5 минут таймер меняет стиль (класс `warning`). Пользователь видит критическое состояние.
- **Экран результатов с разбором ответов** — после аттестации пользователь видит каждый вопрос с иконкой «правильно/неправильно», что создаёт обратную связь для обучения.
- **Переключение «Материалы / Курсы»** в разделе обучения — разграничивает короткие аттестации и структурированные курсы без перегрузки единого списка.
- **Лидерборд с фильтрами по филиалу и должности** — сотрудник может видеть рейтинг как в масштабе всей компании, так и в контексте своей команды. Это мотивирует к соревнованию на реалистичном уровне.
- **Подсветка текущего пользователя в лидерборде** (`current-user` класс) — пользователь мгновенно находит себя в списке.

### ❌ Проблемы

- **Дашборд показывает только одну ближайшую аттестацию.** Если у сотрудника 3 открытые аттестации, он видит только «следующую» — остальные скрыты. Нет сводки: «У вас 2 активные аттестации». Сотрудник может пропустить дедлайн.
- **Точечная навигация не кликабельная.** Сотрудник видит точки вопросов, но не может перейти к конкретному вопросу — только «Далее» по порядку. При большом тесте нельзя вернуться к вопросу, на который хочется изменить ответ.
- **Matching-тип вопроса реализован через `<select>`.** Для мобильного интерфейса Telegram это особенно неудобно: маленькие select-элементы неудобно нажимать пальцем. Нет drag-and-drop или визуального сопоставления карточек.
- **Нет отображения правильных ответов на экране результатов.** Показывается только «правильно/неправильно» для каждого вопроса, но не указывается правильный ответ. Сотрудник не может понять свою ошибку и улучшить знания.
- **Экран «Нет активных аттестаций» не подсказывает, что делать дальше.** Сотрудник видит пустой дашборд без навигации к разделу «Обучение» или «Лидерборд». Dead end в интерфейсе.
- **Страница регистрации (`RegistrationView.vue`) существует в роутере**, несмотря на требование из документации: «Пользователи не должны иметь возможность пройти регистрацию самостоятельно». Это или устаревший маршрут, или потенциальная лазейка.
- **Отсутствует история попыток в профиле.** Пользователь видит сводку (пройдено N, средний балл, бейджей), но не может просмотреть список конкретных попыток с датами и результатами.
- **Таймер предупреждает только при 5 минутах** — без промежуточного предупреждения при половине оставшегося времени. При длинном тесте (60 мин) пользователь может не следить за таймером и потерять 30+ минут работы.
- **Нет индикатора прогресса теории в списке аттестаций.** Если теория обязательна, показывается только текст «Нужно пройти теорию», но не видно, сколько осталось прочитать.

### ⚠️ Точки высокого трения

- **Автозавершение по таймеру не реализовано на серверной стороне** (согласно TASKS.md). Если пользователь закрыл браузер в последние секунды — попытка может зависнуть, а результат будет неопределённым. Это нарушает доверие к системе.
- **Возобновление попытки без фиксированного порядка вопросов** (зафиксированный баг в TASKS.md): при перезаходе вопросы могут перетасоваться. Сотрудник, видевший вопрос под номером 5, теперь видит его под номером 12. Это дезориентирует и нарушает честность аттестации.
- **После провала сотрудник сразу видит кнопку «Пройти ещё раз»** без демонстрации того, что именно пошло не так. Повторная попытка без анализа ошибок — это трата попытки, а не обучение.

### 💡 Рекомендации

- На дашборде добавить счётчик «У вас X открытых аттестаций» со ссылкой на полный список, а не скрывать всё кроме ближайшей.
- Сделать точки вопросов кликабельными для навигации, сохраняя текущий ответ при переходе.
- На экране результатов после провала показывать правильные ответы для вопросов с неверным ответом — это превращает аттестацию в инструмент обучения, а не только оценки.
- Добавить промежуточное предупреждение таймера при достижении половины оставшегося времени (например, при 15 минутах из 30).
- Реализовать drag-and-drop или карточное сопоставление для `matching`-вопросов на мобильном.
- Удалить или защитить маршрут `/registration` если самостоятельная регистрация не предусмотрена.

## 4. Навигация и информационная архитектура

### Административная панель

**Ясность структуры:** Умеренная. Сайдбар содержит 8 пунктов: Дашборд, Пользователи, Аттестации, Курсы, Банк вопросов, Отчёты, Филиалы, Должности, Настройки. Логика группировки не очевидна — «Филиалы» и «Должности» являются справочниками, но стоят рядом с транзакционными модулями.

**Глубина навигации:** Умеренная. Большинство сущностей требует 2–3 клика (список → детали → редактирование). Переход к прогрессу конкретного пользователя в аттестации требует: Аттестации → Детали аттестации → Пользователь — это 3 уровня вложенности без хлебных крошек, что делает навигацию назад неочевидной.

**Поиск ключевых функций:**

- Создать аттестацию: кнопка «+» в шапке раздела — ОК
- Найти пользователя: только через фильтр внутри раздела «Пользователи» — нет глобального поиска
- Просмотреть результаты конкретного сотрудника: нет прямой точки входа (нужно идти через аттестацию или отчёты)
- Настроить геймификацию: Настройки → вкладка «Геймификация» — скрыто, не очевидно для нового менеджера

**Проблема:** Отсутствуют хлебные крошки на глубоких страницах (детали аттестации, прогресс пользователя). Кнопка «Назад» есть, но она навигирует к предыдущей странице браузера, а не к логическому родителю в IA.

### Пользовательское мини-приложение

**Ясность структуры:** Высокая для своей простоты. Нижняя навигация (Bottom Navigation) обеспечивает быстрый доступ к основным разделам.

**Проблема:** Раздел «Обучение» объединяет и аттестации, и курсы под переключателем «Материалы / Курсы». Для нового сотрудника неочевидно, чем «Материалы» отличаются от «Курсов» — нет пояснения при первом входе.

**Отсутствует онбординг.** Новый пользователь, который впервые открыл приложение, видит дашборд без объяснения, что такое аттестация, зачем нужен профиль и как работает система баллов.

## 5. Дизайн-система и UI

### Контекст

Обе части (admin-frontend и frontend) используют **кастомную дизайн-систему на CSS custom properties** (не Shadcn/UI, не стандартный Tailwind). Существует набор переиспользуемых компонентов: `Card`, `Button`, `Input`, `Select`, `Badge`, `Icon`, `Preloader`, `Textarea`.

### ✅ Что работает

- **CSS custom properties** (`--bg-primary`, `--text-secondary`, `--text-primary`) обеспечивают единое изменение темы через одно место. Поддержка тёмной темы реализована на уровне переменных.
- **Компонентный подход** последователен: списки, таблицы, формы используют одни и те же `Card`, `Button`, `Badge` — единообразие на 80%.
- **Иконки через компонент `Icon`** с передачей имени — централизованное управление иконографией, нет прямых SVG по всему коду.
- **Адаптивность через семантические классы** (`hide-mobile`, `show-mobile`) — простой и понятный подход, хотя и не масштабируемый.

### ❌ Где ломается

- **Смешение стилей компонентов и сырых CSS-классов.** В одних местах используется `<Button icon="..." variant="primary">`, в других — `<button class="btn btn-primary btn-full">` напрямую. Это означает два несинхронизированных источника стилей кнопок.
- **Нет design tokens для типографии.** Семантические классы `title-large`, `title-medium`, `body-small` используются, но не везде последовательно. Иерархия заголовков в части представлений выстраивается через `h1`, `h2`, `h3` без привязки к системе.
- **`<select>` в нативном HTML** (в `matching`-вопросах) не стилизован под дизайн-систему — выбивается из общего UI.
- **Отсутствует состояние `empty state` с иконкой и CTA** в большинстве модулей. В `AssessmentsView` пустое состояние — просто `<p>Аттестации не найдены</p>` без предложения действия. Это ведёт в тупик.
- **Нет единой системы уведомлений об ошибках.** В одних формах ошибки показываются через `error`-пропс компонента `Input`, в других — через отдельные строки. Нет глобального toast/snackbar для серверных ошибок.
- **Анимации и переходы непоследовательны.** В некоторых компонентах есть плавные CSS-переходы (сайдбар, прогресс-бар), в других — нет. Это создаёт ощущение неоднородного продукта.

### ⚠️ Риски

- **Масштабируемость кастомного CSS.** При росте команды разработчиков без Tailwind или строгих правил, кастомный CSS рискует превратиться в «глобальный CSS с конфликтами». Уже сейчас схожие классы дублируются между admin-frontend и frontend.
- **Брендинг (жёлтые цвета Панда Пицца) не внедрён.** Интерфейс использует нейтральные цвета. Корпоративный брендбук в системе отсутствует — при масштабировании на партнёров или других клиентов потребует ручной переработки всех компонентов.
- **Нет Storybook или документации компонентов.** При появлении новых разработчиков воспроизведение компонентов потребует изучения каждого файла отдельно.

### 💡 Рекомендации

- Ввести единственный источник стилей для кнопок: либо компонент `Button` везде, либо нативный `button` с CSS-классами — но не оба подхода одновременно.
- Реализовать rich empty states: иконка + заголовок + CTA-кнопка («Создать первую аттестацию»). Это устраняет тупики в навигации.
- Создать глобальный composable/store для уведомлений (toast), объединив все варианты показа ошибок. Composable `useToast.js` уже существует — довести его использование до 100% охвата.
- При внедрении брендинга Панда Пицца использовать CSS custom properties как единую точку изменения цветовой палитры — существующая архитектура это поддерживает.

## 6. Оценка UX-зрелости

## 7. Топ-5 приоритетных исправлений

### 🔴 P0 — Критично (блокирует доверие и корректность данных)

**1. Исправить баги аттестации: возобновление + автозавершение**
Пользователь, перезашедший в аттестацию, видит перетасованные вопросы. Аттестация не завершается автоматически по таймеру. Эти два бага делают результаты аттестаций недостоверными. Без этого любая аналитика по успеваемости некорректна.

**2. Исправить баг оценки matching-вопросов**
Частично правильные ответы на вопросы сопоставления оцениваются неверно. Все аттестации, содержащие такие вопросы, выдают ошибочные баллы. Это может влиять на решения об аттестации сотрудников.

### 🟠 P1 — Высокий приоритет (значительно влияет на использование)

**3. Добавить серверную пагинацию во все списки**
Без неё система неработоспособна при реальной нагрузке (500+ пользователей, 200+ вопросов). При загрузке всех данных разом интерфейс тормозит, а API перегружается. Реализовать: `?page=1&limit=25` + отображение «Показано X из N».

**4. Устранить дублирование подхода к кнопкам и унифицировать empty states**
Два источника стилей (`<Button>` компонент vs `.btn` классы) создают риск рассинхронизации при изменении дизайна. Унифицировать и добавить rich empty states с CTA — это устраняет «тупики» в навигации, которые сейчас гарантированно встречает каждый новый пользователь.

### 🟡 P2 — Средний приоритет (улучшает UX без блокировки)

**5. Добавить «обзор с правильными ответами» на экране результатов после провала**
Сотрудник, провалившийся на аттестации, не получает никакой обучающей ценности — только «неправильно». Показ правильных ответов превращает повторную попытку из «угадывания» в осознанное обучение. Это напрямую влияет на эффективность системы как инструмента развития персонала, а не только проверки.
