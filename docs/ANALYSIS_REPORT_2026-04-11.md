# Анализ проекта assessment

Дата: 2026-04-11

## 1. Что было изучено

- `docs/features.md` (планы развития продукта).
- `docs/AUDIT_REPORT.md` (предыдущий аудит, дата 2025-02-14).
- Backend: маршруты, middleware, контроллеры, модели, сервисы, миграции, конфиги.
- Frontend (MiniApp): роутинг, auth-поток, регистрация, процесс аттестаций.
- Admin frontend: auth, хранение токенов, права модулей, WebSocket.
- Bot: текущий функционал и интеграции.
- CI/CD: workflow деплоя backend.

## 2. Текущее состояние по модулям

## Backend

- Сильные стороны:
  - Есть базовая модульная архитектура (routes/controllers/models/services).
  - Используются prepared statements (`execute`/`query` с параметрами).
  - Реализованы роли, модульные права, геймификация, теория, попытки аттестаций.
  - Есть WebSocket-слой и централизованный `errorHandler`.

- Проблемные точки:
  - JWT-секреты имеют небезопасные fallback-значения.
  - CORS в режиме `origin: true` при пустом `ALLOWED_ORIGINS`.
  - Аудит формально вызывается, но фактически не сохраняется.
  - Несогласованность правил доступа менеджера (по коду и комментариям).
  - Обнаружены логические/SQL дефекты в модели аттестаций.

## MiniApp frontend

- Сильные стороны:
  - Нормальный поток Telegram initData.
  - Есть сценарии аттестаций, теории, лидерборда, профиля.

- Проблемные точки:
  - Часть ключевых view не на `<script setup>` (несоответствие внутренним правилам).
  - Много debug `console.log` в runtime.
  - Ряд требований `features.md` (курсы, мультиязычность, расширенный контент) пока не реализованы.

## Admin frontend

- Сильные стороны:
  - В целом современный стек и в большинстве компонентов `<script setup>`.
  - Реализованы модульные проверки доступа на уровне роутера и API.

- Проблемные точки:
  - Access token хранится в `localStorage` (повышенный риск при XSS).
  - Есть расхождения между дефолтными модулями в UI и backend.
  - Много отладочных логов, часть можно убирать из production.

## Bot

- Текущий бот минимальный (`/start` + кнопка открытия MiniApp).
- По `features.md` нужен переход к полноценному notification/onboarding/FAQ-боту.

## Миграции и БД

- Плюсы:
  - Хорошая базовая схема: FK, индексы, `utf8mb4_unicode_ci`.

- Риски:
  - Нет таблицы истории миграций (`schema_migrations`).
  - Конфликтующие номера миграций (`002_*` в двух файлах).
  - Runner миграций хрупко делит SQL по `;` + перенос строки.
  - Есть `SELECT *` в ряде запросов.

## 3. Проверка соответствия plans в features.md

## Уже есть (полностью/частично)

- Геймификация, уровни, бейджи, лидерборд (частично гибко, но требует доработок).
- Теория по аттестациям (text/video/link, версии/draft/publish).
- Модульные права доступа для админ-панели.
- Инвайты и привязка пользователя к Telegram.

## Пока не закрыто

- Полноценная система уведомлений в боте (user/admin/manager, deep-link кнопки).
- Реальный аудит с отправкой в Telegram thread.
- Курсы с подтемами и итоговой аттестацией курса.
- Мультиязычность контента и интерфейса.
- Расширенный контент теории (image/audio/iframe как отдельные типы).
- Эталонные текстовые ответы по триггерам/нестрогому совпадению.
- GiftShop (магазин за баллы).
- Конструктор отчетов (OLAP-подобный сценарий).

## 4. Критичные найденные проблемы (сводка)

1. Секреты JWT с fallback.
2. CORS `origin: true` + `credentials: true`.
3. Аудит не пишет события фактически.
4. У менеджера нет полной изоляции по филиалу в ряде операций.
5. Дефекты в логике аттестаций:
   - ошибочная ветка `matching`;
   - некорректное имя таблицы в запросе;
   - неполная реализация фиксированного порядка вопросов/ответов попытки.

## 5. Архитектурные рекомендации

- Вынести безопасность в отдельный baseline:
  - `helmet`, rate-limit, CSRF для cookie flow, строгий CORS.
- Перевести auth-модель админки:
  - refresh в HttpOnly cookie,
  - access token в памяти, без хранения в `localStorage`.
- Ввести единый policy-слой авторизации:
  - филиал/роль/модуль в одном месте, без дублирования логики.
- Добавить data-migration discipline:
  - `schema_migrations`,
  - уникальные номера,
  - транзакционность/идемпотентность.
- Ввести quality gates:
  - lint/test/build в CI до deploy.

## 6. Заключение

Проект имеет сильную функциональную базу и уже содержит важные доменные блоки, но в текущем виде требует фазы стабилизации (безопасность + доступы + исправление дефектов в аттестациях) до активной реализации крупных фич из `features.md`. Наиболее правильный путь: сначала закрыть P0/P1 по надежности и безопасности, затем идти в продуктовые расширения (курсы, уведомления, мультиязычность, GiftShop, конструктор отчетов).
