## 1. EXECUTIVE SUMMARY

### Общий обзор проекта

Внутренняя LMS-платформа для обучения и аттестации сотрудников (ресторанная/сервисная сеть). Включает:

- **Backend** — Node.js / Express, MySQL, Redis, BullMQ
- **Admin-Frontend** — Vue 3, Pinia, TailwindCSS (управление курсами, пользователями, геймификацией)
- **Frontend** — Vue 3, Pinia (Telegram MiniApp + MAX MiniApp для сотрудников)
- **Bot** — Telegraf (Telegram) + поддержка MAX-платформы

**Бизнес-домен:** корпоративное обучение, аттестации, геймификация, сертификаты, ролевая модель доступа.

### Общая оценка качества архитектуры

Проект демонстрирует **зрелый подход к структурированию** для монолитного Node.js-приложения. Выбранная многослойная архитектура (Routes → Controller → Service → Repository) последовательно соблюдается в большинстве модулей. Вместе с тем присутствуют системные проблемы: неполная реализация RBAC, отсутствие API-версионирования, технический долг в блоке безопасности.

### Уровень технической зрелости

**Средний+** (Intermediate-Advanced). Проект вышел за рамки прототипа, имеет продуманную доменную модель, тесты, миграции и runbook. Не достиг enterprise-уровня из-за отсутствия CI/CD, мониторинга и незавершённой RBAC-системы.

### Оценка рисков

| Категория риска  | Уровень     | Комментарий                                                                              |
| ---------------- | ----------- | ---------------------------------------------------------------------------------------- |
| Безопасность     | MEDIUM-HIGH | BOT_TOKEN как Bearer-токен для internal API, refresh-токен в БД без ротации              |
| Масштабируемость | MEDIUM      | Монолит, нет горизонтального масштабирования, in-memory метрики                          |
| Надёжность       | MEDIUM      | Redis опционален (fallback), очередь геймификации теряет события при недоступности Redis |
| Технический долг | MEDIUM      | Незакрытый статус `completed_locked`, legacy CRUD auth flag в production                 |
| Инфраструктура   | HIGH        | Нет Docker/CI/CD, нет резервного копирования, нет observability                          |

### Итоговая оценка проекта

**6.5 / 10**

## 2. АНАЛИЗ СТРУКТУРЫ ПРОЕКТА

### 2.1 Организация репозитория

```
assessment/
├── backend/          # Express API + бизнес-логика
├── admin-frontend/   # Vue 3 SPA для администраторов
├── frontend/         # Vue 3 MiniApp (Telegram / MAX)
├── bot/              # Telegraf-бот (отдельное приложение)
├── docs/             # Документация, инжиниринговые регламенты
├── logs/             # Логи (корневые — ошибка структуры)
├── uploads/          # Загружаемые файлы
└── ROADMAP.md / TASKS.md / AGENTS.md
```

**Проблемы:**

- Директория `logs/` в корне репозитория рядом с `backend/logs/` — дублирование и путаница.
- Директория `uploads/` в корне дублируется с `backend/uploads/` — неясно, какая используется.
- Проект не является монорепо с явной конфигурацией (нет `pnpm-workspace.yaml` / `nx.json`), но содержит 3 независимых `package.json` — риск рассинхронизации версий зависимостей.

### 2.2 Структура Backend-модулей

Соответствует задокументированному стандарту `Routes → Controller → Service → Repository`. Большинство модулей (`courses`, `auth`, `assessment`, `gamification`) следуют этому паттерну.

**Сильные стороны:**

- Чёткое разделение ответственности по слоям
- Публичный `index.js` в каждом модуле
- Отдельные `admin/` и `public/` подмодули для изоляции прав доступа
- Наличие маппёров (`courses.mappers.js`) для трансформации DTO

**Проблемы:**

| Проблема                                                                                                                       | Расположение                        | Серьёзность |
| ------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------- | ----------- |
| `courses/admin/service.js` содержит строки «урс не найден» вместо «Курс не найден» — признак незамеченной деградации кодировки | `modules/courses/admin/service.js`  | LOW         |
| Репозиторий `courses.repository.js` выполняет валидацию публикации (`validatePublicationIntegrity`) — нарушение SRP            | `courses.repository.js`             | MEDIUM      |
| `courseProgressService.js` напрямую импортирует `coursesRepo` и `progressRepo` — частичное нарушение изоляции слоя сервиса     | `services/courseProgressService.js` | LOW         |
| `bot/src/index.js` реализует HTTP-клиент вручную (нативный `http`/`https`) вместо использования общего HTTP-клиента            | `bot/src/index.js`                  | MEDIUM      |

### 2.3 Размер файлов

- `courses.repository.js` — предположительно > 500 строк (признаки: много функций, несколько доменных операций). Требует декомпозиции на `courses.read.repository.js` / `courses.write.repository.js`.
- `courseProgressService.js` — содержит полный flow `startCourse`, `handleTopicMaterialViewed` и другие — кандидат на разбивку.

### 2.4 Зависимости

**Backend:** Все используемые пакеты обоснованы. Дублирований нет. `joi` и `express-validator` существуют параллельно — стоит стандартизировать на одном.

**Frontend / Admin:** Набор зависимостей рациональный. `chart.js` + `vue-chartjs`, `tiptap`, `vuedraggable` — все оправданы функциональностью.

## 3. АУДИТ КАЧЕСТВА КОДА

### 3.1 Читаемость и сопровождаемость

**Плюсы:**

- Код написан последовательно, комментарии на русском языке
- Функции небольшие, однофункциональные
- Именование переменных понятное (`topicsBySectionId`, `syncSectionCompletionAfterTopicProgress`)
- Использование `async/await` без смешения с `.then()`-цепочками

**Минусы:**

- В `bot/src/index.js` самописный HTTP-клиент с дублированием кода для `backendGet` и `backendPatch` — нарушение DRY
- В `botService.js` оставлены `console.log` в production-коде (строки с `[botService.getUserStatusByTelegramId]`) — нарушение стандарта

### 3.2 Обработка ошибок

**Плюсы:**

- Единый `errorHandler` middleware с корректной классификацией (500 / 503 для транзиентных DB-ошибок)
- Транзакции с `try/catch/finally` и `connection.rollback()` + `connection.release()` везде где это найдено
- Сервисы создают ошибки с `.status` кодом — корректный pattern

**Минусы:**

- `eventsRepo.insertCourseEvent` вызывается **после** `commit()` без `await` и без обработки ошибок — потеря событий при сбое
- В `bot/src/index.js` функция `getUserStatus` не обрабатывает таймаут HTTP-запроса — риск зависания при недоступном backend

### 3.3 Валидация входных данных

- Используется `express-validator` для части маршрутов
- `joi` присутствует как зависимость — видимо, используется в другой части
- Нет единого стандарта — часть маршрутов может не иметь валидации

### 3.4 State Management (Frontend)

- Pinia — современный выбор, корректное использование
- Telegram-стор (`telegram.js`) содержит сложную логику парсинга `initData` — оправдано спецификой платформы, код хорошо изолирован
- `INIT_DATA_MAX_AGE_SECONDS = 5 * 60` на клиенте — дублирует серверную проверку (24 часа), несоответствие является риском UX: пользователя могут выкинуть из сессии через 5 минут без видимой причины

### 3.5 Async-операции

- `gamificationQueueService.js` корректно использует BullMQ с `concurrency: 10`, retry 3 раза с экспоненциальным backoff
- Ленивая инициализация `worker` и `queue` через `ensure*()` — приемлемый паттерн

## 4. АНАЛИЗ TELEGRAM-ПЛАТФОРМЫ

### 4.1 Архитектура бота

Бот реализован как **отдельное Node.js-приложение** (`bot/`) на базе `telegraf`. Взаимодействие с backend происходит через внутренний HTTP API (`/api/bot/internal/*`), защищённый `verifyBotToken` middleware.

**Плюсы:**

- Изоляция бота в отдельный процесс — корректное решение
- Унифицированный `backendGet`/`backendPatch` для запросов к API
- Обработка onboarding flow с configurable-текстами из `system_settings`

**Проблемы:**

| Проблема                              | Серьёзность | Описание                                                                                                     |
| ------------------------------------- | ----------- | ------------------------------------------------------------------------------------------------------------ |
| Самописный HTTP-клиент без таймаутов  | HIGH        | Запрос к backend может висеть неограниченно, блокируя обработчик бота                                        |
| Отсутствие FSM (Finite State Machine) | MEDIUM      | Нет явного управления состоянием диалога — при росте команд код усложнится                                   |
| Нет rate-limiting на уровне бота      | MEDIUM      | Telegram ограничивает 30 сообщений/сек к группам и 1 сообщение/сек к пользователю; бот не контролирует flood |
| Нет webhook — используется polling    | MEDIUM      | В production polling неэффективен, webhook предпочтителен                                                    |
| Отсутствие session-management         | LOW         | Нет персистентных сессий для multi-step диалогов                                                             |

### 4.2 Безопасность initData

`telegramAuthService.js` — корректная реализация HMAC-SHA256 валидации Telegram initData:

- Используется `crypto.timingSafeEqual` — защита от timing-атак ✅
- Проверка `auth_date` (не старше 24 часов) ✅
- Нормализация URL-encoding ✅

**Проблема:** отсутствие проверки `auth_date` на значение из **будущего** (защита от replay-атак с предсгенерированными токенами).

### 4.3 Внутренний API бота

`verifyBotToken` проверяет `Authorization: Bearer <BOT_TOKEN>`. **Критическая проблема:** BOT_TOKEN — это токен Telegram-бота, предназначенный для Telegram API, а не для внутренней аутентификации. Компрометация инфраструктуры (логи, env) одновременно открывает и Telegram-бота, и internal API.

## 5. АНАЛИЗ MAX-ПЛАТФОРМЫ

### 5.1 Качество интеграции

MAX-платформа поддержана на уровне `maxAuthService.js`, который является форком `telegramAuthService.js` с адаптацией под `MAX_BOT_TOKEN`.

**Плюсы:**

- Единый `verifyInitData` middleware с автоопределением платформы через заголовок `x-client-platform`
- `normalizeInitData` в MAX-сервисе обрабатывает URL-encoded строки — практически важная деталь
- `maxBotToken` fallback на `botToken` в `config/env.js` — удобно для dev-среды

**Проблемы:**

| Проблема                                           | Серьёзность | Описание                                                                                                                                                       |
| -------------------------------------------------- | ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Логика выбора платформы в `verifyInitData` хрупкая | MEDIUM      | `platform = requestedPlatform \|\| (maxInitData ? "max" : "telegram")` — при отсутствии заголовка и наличии `initData` в body может выбрать неверную платформу |
| Нет проверки `auth_date` в MAX initData            | HIGH        | В отличие от Telegram-сервиса, `maxAuthService.js` не проверяет срок жизни токена (нет `auth_date` validation)                                                 |
| Один `refresh_token` для обеих платформ            | MEDIUM      | Отсутствует разграничение сессий по платформе                                                                                                                  |
| Документация MAX-специфики минимальна              | LOW         | Runbook есть, но детального описания отличий платформ нет                                                                                                      |

### 5.2 Кросс-платформенная согласованность

Frontend использует `getClientPlatform()` и `setTelegramRuntimeState()` — абстракция есть, но именование (`telegram*` для общей платформы) создаёт путаницу при поддержке.

## 6. АНАЛИЗ БАЗЫ ДАННЫХ

### 6.1 Общая структура

35 миграций. Используется MySQL (InnoDB, utf8mb4). Миграции выполняются через `database/runMigrations.js`.

**Основные сущности:**

- `users`, `roles`, `branches`, `positions`
- `assessments`, `assessment_attempts`, `assessment_questions`
- `courses`, `course_sections`, `course_topics`, `course_section_user_progress`
- `gamification_rules`, `user_points_history`, `badges`
- `invitations`, `certificates`
- `audit_logs`, `system_settings`

### 6.2 Качество схемы

**Плюсы:**

- `utf8mb4` для всех таблиц — корректная поддержка эмодзи
- Внешние ключи с `ON DELETE CASCADE` где уместно
- Миграция 013 добавляет составные индексы для часто используемых запросов
- `UNSIGNED AUTO_INCREMENT` для PK — эффективно

**Проблемы:**

| Проблема                                                                                                           | Серьёзность | Таблица                      |
| ------------------------------------------------------------------------------------------------------------------ | ----------- | ---------------------------- |
| Нумерация миграций 031 встречается 3 раза (031_add_company_settings, 031_create_audit_logs, 031_soft_delete_users) | HIGH        | migrations/                  |
| Нет миграции для таблицы `course_drafts` — упоминается в коде, но миграция не выделена явно                        | MEDIUM      | —                            |
| `users.refresh_token TEXT` — хранение refresh_token в БД без хеширования                                           | HIGH        | users                        |
| `users.telegram_id VARCHAR(32)` — достаточно, но не индексировано отдельно                                         | MEDIUM      | users                        |
| Статус `completed_locked` в курсах не реализован на уровне БД                                                      | HIGH        | course_section_user_progress |
| Отсутствуют индексы на `course_section_user_progress(user_id, course_id)`                                          | MEDIUM      | course_section_user_progress |

### 6.3 ORM и запросы

- Используется `mysql2` с `pool.execute()` — параметризованные запросы, нет SQL-инъекций ✅
- Транзакции (`beginTransaction / commit / rollback`) применяются корректно
- Паттерн `connection.execute()` с явной передачей `connection` в репозитории — правильное решение для транзакций

**Потенциальные N+1:**

```javascript
// courses.repository.js
const sectionsWithTopics = await Promise.all(
  sections.map(async (section) => ({
    ...section,
    topics: await listTopicsBySectionId(section.id),
  })),
);
```

Для N секций выполняется N+1 запросов. При курсах с 10+ секциями — заметная нагрузка. Требует JOIN-запроса или батч-загрузки.

## 7. АНАЛИЗ API

### 7.1 Соответствие REST-стандартам

Проект имеет задокументированные API Guidelines (`docs/engineering/api-guidelines.md`), предписывающие `/api/v{n}/` формат. **Фактически API работает без версии** (`/api/health`, `/api/auth/*`). Это нарушение собственного стандарта, критично при breaking changes.

**Типичные маршруты:**

```
POST /api/auth/register
GET  /api/courses/:id
POST /api/admin/courses/:id/upload-cover
POST /api/admin/courses/:courseId/topics/:topicId/start
```

**Проблемы:**

| Проблема                                                              | Серьёзность |
| --------------------------------------------------------------------- | ----------- |
| Отсутствие версионирования API (`/api/v1/`)                           | HIGH        |
| Маршруты с глаголами (`/start`, `/complete`, `/upload-cover`)         | MEDIUM      |
| Пагинация реализована не во всех list-endpoints                       | MEDIUM      |
| Нет единого формата ошибок (иногда `{ error }`, иногда `{ message }`) | MEDIUM      |

### 7.2 Аутентификация и авторизация

- JWT в `Authorization: Bearer` заголовке — стандартно
- `verifyJWT` middleware лаконичен и корректен
- `requirePermission` middleware реализует RBAC, но **`legacyCrudAuthEnabled`** flag в production обходит granular-проверки для `manager` и `superadmin` — это временное решение с риском застрять навсегда

### 7.3 Rate Limiting

- `adminGlobalLimiter` — глобальный лимит для `/admin/*`
- `verifyCertificateLimiter` — специфичный лимит
- **Нет rate limiting для публичных auth-маршрутов** (`/api/auth/register`, `/api/auth/status`) — риск брутфорса

## 8. АУДИТ БЕЗОПАСНОСТИ

### 8.1 Критические проблемы

| #   | Проблема                                                            | Тип               | Серьёзность |
| --- | ------------------------------------------------------------------- | ----------------- | ----------- |
| 1   | `refresh_token` хранится в `users` таблице в открытом виде          | Хранение секретов | CRITICAL    |
| 2   | BOT_TOKEN используется как Bearer-токен для internal API            | Аутентификация    | HIGH        |
| 3   | Нет проверки `auth_date` на значение из будущего                    | Replay attack     | HIGH        |
| 4   | MAX initData не проверяет `auth_date` (нет срока жизни)             | Token expiry      | HIGH        |
| 5   | `console.log` в `botService.js` логирует `telegramId` пользователей | Data leakage      | MEDIUM      |
| 6   | Нет CSRF-защиты для cookie-based сессий (если применяется)          | CSRF              | MEDIUM      |

### 8.2 Управление секретами

**Плюсы:**

- `config/env.js` выбрасывает ошибку при старте, если `JWT_SECRET` или `JWT_REFRESH_SECRET` отсутствуют
- В production проверяется `ALLOWED_ORIGINS`
- Явный список `requiredVars` с предупреждением при отсутствии

**Минусы:**

- `maxBotToken` fallback на `botToken` — в prod это может привести к использованию неверного токена без явной ошибки
- Отсутствие rotation strategy для JWT-секретов и BOT_TOKEN

### 8.3 Безопасность загрузок файлов

- Используется `multer` — библиотека с известными уязвимостями при неправильной настройке
- Сертификаты защищены `verifyJWT` middleware ✅
- Остальные `uploads/` доступны без аутентификации — возможна утечка через прямые URL

### 8.4 Helmet и CSP

`helmet` настроен с `contentSecurityPolicy`. CSP содержит только базовые директивы без `img-src`, `script-src`, `connect-src` — возможны XSS через подзагружаемые ресурсы.

### 8.5 SQL-инъекции

Все найденные запросы используют `pool.execute(sql, params)` с параметрами. SQL-инъекций не обнаружено ✅.

### 8.6 Зависимости

- `multer: ^1.4.5-lts.2` — LTS-ветка, поддерживается
- `ffmpeg-static: ^5.3.0` — добавляет ~300MB к образу, использование не очевидно из кода
- `moment-timezone: ^0.6.0` — устаревшая библиотека, рекомендован переход на `date-fns` или `luxon`

## 9. АНАЛИЗ ПРОИЗВОДИТЕЛЬНОСТИ И МАСШТАБИРУЕМОСТИ

### 9.1 Узкие места

| Проблема                                                                           | Место                         | Влияние                                     |
| ---------------------------------------------------------------------------------- | ----------------------------- | ------------------------------------------- |
| N+1 при загрузке курсов с секциями и темами                                        | `courses.repository.js`       | Высокое при > 10 секций                     |
| Метрики хранятся in-memory (`requestDurations[]` — 500 элементов)                  | `metricsService.js`           | Сбрасываются при перезапуске, нет агрегации |
| Нет кеширования списка курсов                                                      | `courses.repository.js`       | Каждый запрос — полный SELECT               |
| `Promise.all()` на параллельные DB-запросы без пула соединений с лимитом           | Множество мест                | Риск исчерпания пула при высокой нагрузке   |
| Геймификация обрабатывается синхронно при ответе на вопрос (если Redis недоступен) | `gamificationQueueService.js` | Увеличивает latency ответа                  |

### 9.2 Кеширование

`CacheService` реализован поверх Redis с graceful-деградацией при недоступном Redis. Это хорошее решение. Однако кеш **фактически не используется** для курсов и других часто запрашиваемых данных — только сервис есть, применения нет.

### 9.3 Горизонтальное масштабирование

Текущая архитектура **не готова к горизонтальному масштабированию** из-за:

- In-memory метрик (не shared)
- Однопроцессного scheduler уведомлений (`notificationScheduler.js`)
- Stateful WebSocket-сессий без Redis adapter

### 9.4 Оценка нагрузки

При текущей архитектуре система способна обслуживать **500–2000 активных пользователей** одновременно (оценка по типовой нагрузке Express + MySQL без кеша). Для масштабирования выше потребуется:

1. Redis-based session/cache
2. Горизонтальное масштабирование через PM2 cluster или Kubernetes
3. Read-replika для MySQL

### Готовность к production

Система **может работать в production** при условии устранения критических проблем (Фаза 1). До этого риски компрометации сессий (refresh_token) и несанкционированного доступа к internal API являются блокерами.

### Готовность к масштабированию

**Не готова.** Требуется: Docker, горизонтальное масштабирование, Redis adapter для WebSocket, Prometheus-метрики. Оценка: 3–4 месяца до production-ready масштабируемой архитектуры.

### Основные риски

1. **Безопасность сессий** — refresh_token без хеширования
2. **Internal API** — защищён тем же токеном, что и Telegram-бот
3. **Отсутствие DevOps** — ручной деплой, нет резервного копирования
4. **Технический долг RBAC** — legacy-флаг обходит всю систему прав

### Основные сильные стороны

1. **Чёткая многослойная архитектура** — последовательно соблюдается во всех модулях
2. **Двойная платформа** — качественная поддержка Telegram + MAX с едиными middleware
3. **Транзакционная целостность** — все критические операции в транзакциях
4. **Геймификация через очередь** — async через BullMQ, правила из БД
5. **Инженерная документация** — архитектурные регламенты, runbook, API guidelines
6. **Тесты** — покрыты ключевые бизнес-сценарии (курсы, аттестации, геймификация)
7. **Безопасный HMAC** — `timingSafeEqual` для валидации initData

### Инвестиционная привлекательность

**Средняя-высокая.** Продукт имеет чёткий бизнес-домен, работающую функциональность и хорошую архитектурную базу. Технический долг управляем и оценивается в 2–3 месяца работы команды для полного устранения. Платформа готова к росту аудитории при закрытии Фаз 1–2.

5. **Прогресс курса** — линейный прогресс-бар на уровне курса и секции с процентами
6. **Адаптивный matching** — тип вопроса "сопоставление" имеет два режима: mobile (chips) и desktop (select)

### Ключевые UX-слабые стороны

1. **Глубина навигации** — 4 уровня вложенности (Курс → Секция → Тема → Подтема) без breadcrumbs
2. **Кнопка уведомлений** — иконка `Bell` на dashboard присутствует, но функционально не подключена (dead UI element)
3. **Бот без FSM** — отсутствие состояний диалога делает multi-step флоу хаотичными при росте сценариев
4. **Регистрация с одной кнопкой** — нет подсказки что делать если телефон не найден до нажатия кнопки
5. **Онбординг как модальный overlay** — закрывает весь контент, не показывает, что ждёт внутри
6. **Admin без bulk-действий** — нет массового назначения курсов; управление пользователями по одному

### Риски удержания пользователей

- Вынужденный logout через 5 минут (клиентский `INIT_DATA_MAX_AGE_SECONDS`) без понятной причины
- Отсутствие push-уведомлений о прогрессе и назначенных курсах через бот
- Нет offline graceful degradation — при потере сети MiniApp полностью ломается
- Нет "возвращающего" механизма: бот не напоминает о незаконченных курсах

### Конверсионные узкие места

- Регистрация через подтверждение номера телефона — высокий барьер для пользователей, отказывающихся делиться контактом
- Онбординг-оверлей блокирует доступ к контенту до нажатия кнопки — нет preview
- Финальная аттестация скрыта за завершением **всех** тем — нет промежуточных точек вовлечения

## 19. АНАЛИЗ ИНФОРМАЦИОННОЙ АРХИТЕКТУРЫ

### 19.1 Структура навигации MiniApp

MiniApp использует bottom navigation с пятью разделами: Главная, Аттестации, Курсы (отсутствует явный пункт!), Лидерборд, Профиль. Раздел "Курсы" недоступен из нижней навигации напрямую — курсы открываются через Dashboard или через deep link.

**Проблемы:**

| Проблема | Описание | Влияние |
|---|---|---|
| Курсы не в bottom nav | Основной контент LMS недостижим из главной навигации | Высокое — пользователь не знает где искать курсы |
| Аттестации занимают отдельный пункт nav | Аттестации — подчинённая сущность курса, но выделена наравне с главной страницей | Среднее — нарушение иерархии IA |
| Лидерборд в основной nav | Второстепенная gamification-функция занимает один из 4 слотов | Низкое — вытесняет более важный контент |

### 19.2 Глубина навигации

Путь до учебного контента:
```
Dashboard → [кнопка "Продолжить"] → CourseDetailsView → SectionList → TopicView → SubtopicView → (контент)
```

Это **5 экранов** до достижения учебного материала от главного экрана. При линейном прохождении каждый раз нужно проходить весь путь заново. Нет "быстрого перехода к последнему месту".

**Рекомендация:** Кнопка "Продолжить" на Dashboard должна вести непосредственно к SubtopicView (последней незавершённой подтеме), а не к CourseDetailsView.

### 19.3 Структура Admin Panel

Admin Panel организована по доменным модулям: Курсы, Пользователи, Аттестации, Геймификация, Отчёты, Настройки. Структура логична, но:

- Нет глобального поиска по всем сущностям
- Нет "быстрых действий" (создать курс, назначить пользователя) без входа в конкретный раздел
- Breadcrumbs в редакторе курсов отсутствуют — непонятно где находишься внутри 4-шагового wizard

### 19.4 Обнаруживаемость функций

| Функция | Обнаруживаемость | Проблема |
|---|---|---|
| Достижения | Низкая | Спрятаны в `/profile/achievements`, не видны из профиля без скролла |
| История обучения | Низкая | `/profile/history` — нет прямой ссылки из dashboard |
| Теория аттестации | Низкая | Экран `/assessment/:id/theory` есть, но кнопка найти трудно |
| Сертификаты | Средняя | Отдельный пункт nav есть, но связь с конкретным курсом неочевидна |

### 19.5 Ментальные модели пользователя

Корпоративный LMS-пользователь ожидает:
- "Мои курсы" как отправная точка — в текущей IA это называется Dashboard, что менее конкретно
- Прогресс как главный ориентир — есть на Dashboard в виде 3 статистических карточек, но без визуального прогресс-бара по всем курсам
- Чёткое разграничение "учёба" vs "тест" — в текущей IA они соединены, что создаёт когнитивную нагрузку

## 20. АНАЛИЗ ПОЛЬЗОВАТЕЛЬСКИХ ФЛОУ

### 20.1 Флоу студента: регистрация и онбординг

```
[Telegram/MAX] → Открыть MiniApp → RegistrationView → [Кнопка "Подтвердить номер"] 
→ Telegram requestContact → Backend auth → onboarding overlay → Dashboard
```

**Фрикции:**

1. **Единственная CTA без контекста** — кнопка "Подтвердить номер телефона" — единственный элемент на экране. Пользователь не знает, что произойдёт если номер не найден, до нажатия.
2. **Нет fallback-объяснения** — текст "Если номер не найден, понадобится персональная ссылка" показан *ниже* кнопки мелким шрифтом, после ошибки. Пользователь видит ошибку, а не инструкцию.
3. **Telegram requestContact** — системный диалог Telegram для получения контакта вызывает тревогу у пользователей, незнакомых с механизмом. Нет объяснения зачем нужен контакт.
4. **Онбординг-оверлей** — после авторизации статичный overlay с 3 пунктами списка и одной кнопкой. Нет визуального превью контента, нет персонализации.

**Рекомендации:**
- Добавить inline-объяснение "Ваш телефон нужен только для идентификации" перед кнопкой
- Показать ошибку с конкретной CTA "Запросить доступ у администратора → [кнопка копирования]"
- Онбординг заменить на tour с 2-3 шагами, показывающий реальный интерфейс

### 20.2 Флоу студента: прохождение курса

```
Dashboard → [кнопка "Продолжить"] → CourseDetailsView (список секций) 
→ TopicView (список подтем + кнопки) → SubtopicView (контент + таймер) → [Завершить]
→ Следующая подтема или возврат к TopicView
```

**Фрикции:**

1. **Лишний экран CourseDetailsView** — при повторном входе пользователь попадает на список секций, а не туда, где остановился. 4 клика вместо 1 для возобновления.
2. **Нет "быстрого перехода"** — кнопка на Dashboard ведёт на CourseDetailsView, не на SubtopicView с последней незавершённой подтемой.
3. **TopicView дублирует структуру** — одновременно показывает список тем и является промежуточным экраном. Пользователю неясно, нужно ли открывать подтему или начинать тему целиком.
4. **Таймер чтения в SubtopicView** — таймер чтения скрыт от пользователя (синхронизируется с сервером), но блокирует завершение. Пользователь не видит, почему кнопка "Завершить" неактивна.
5. **Навигация через Browser back** — нет breadcrumbs, нет кнопки "К списку", только системный back через `useBackButton`.

### 20.3 Флоу студента: аттестация

```
AssessmentsView (список) → AssessmentTheoryView (опционально) 
→ AssessmentProcessView (вопросы) → AssessmentResultsView
```

**Сильные стороны:**
- Прогресс-бар с процентом и счётчиком вопросов
- Таймер с предупреждением при < 5 минут (красный класс)
- Адаптивный интерфейс для matching-вопросов (chips на mobile)

**Фрикции:**
1. **Нет возможности вернуться к вопросу** — вопросы листаются линейно, нет навигации "Назад к вопросу N". При случайном нажатии пользователь теряет выбор.
2. **Matching на мобильном** — UI с chip-кнопками для каждого варианта создаёт длинный скролл при 5+ парах. Кнопка "Сбросить выбор" слабо заметна.
3. **Text-answer вопросы** — textarea без count символов и без подсказки о минимальной длине. Пользователь не знает, достаточен ли его ответ.
4. **Нет черновика** — при закрытии MiniApp или потере сети прогресс аттестации теряется, нет auto-save.

### 20.4 Флоу администратора: создание курса

```
CoursesView → [Создать курс] → CourseEditorView 
→ Шаг 1 (Основная информация) → Шаг 2 (Структура) → Шаг 3 (Тесты) → Шаг 4 (Публикация)
```

**Сильные стороны:**
- 4-шаговый wizard с явным stepper
- Drag-and-drop для переупорядочивания секций и тем
- Tiptap rich text editor для контента материалов
- Валидация перед публикацией с выводом ошибок

**Фрикции:**
1. **Wizard без авто-сохранения шага** — при случайном закрытии вкладки работа теряется. Нет индикатора "сохранено" vs "не сохранено".
2. **Шаг 2 (Структура) — перегружен** — редактор передаёт 40+ props в `CourseEditorStructureStep`. Экран содержит одновременно список секций, список тем, форму новой секции, форму новой темы и редактор материала. Когнитивная перегрузка.
3. **Material Editor внутри Структуры** — редактор материала (Tiptap) открывается как панель внутри того же шага, не как отдельный фокусный экран. При узком экране место для редактора критически мало.
4. **Нет предпросмотра курса** — администратор не может увидеть, как курс выглядит с точки зрения студента, не выходя из редактора.

### 20.5 Флоу учителя / менеджера

Отдельного интерфейса учителя не выделено: менеджер использует Admin Panel. Это архитектурное решение создаёт проблему:

- Менеджер видит все административные инструменты, которые ему не нужны
- Нет упрощённого "учительского" режима для проверки домашних заданий и мониторинга прогресса студентов
- Проверка домашних заданий (если реализована) не имеет выделенного workflow

## 21. АНАЛИЗ TELEGRAM UX

### 21.1 Качество бот-интерфейса

Telegram-бот реализован на Telegraf с полlingом. Из кода `bot/src/index.js` видно, что бот обрабатывает команды (`/start`) и текстовые сообщения с открытием MiniApp через inline keyboard.

**Проблемы качества бот-диалога:**

| Проблема | Серьёзность | Описание |
|---|---|---|
| Нет FSM — отсутствие состояний диалога | HIGH | При росте сценариев (регистрация через бот, восстановление доступа, поддержка) код превратится в спагетти без управления состоянием |
| Polling вместо webhook | MEDIUM | Задержка ответа бота 1–3 секунды вместо мгновенного ответа через webhook. Пользователь воспринимает бота как "медленного" |
| Нет session management | MEDIUM | Нет персистентных сессий — каждое взаимодействие обрабатывается stateless. Multi-step диалоги (например, регистрация по шагам) невозможны без внешнего хранилища |
| Повторный рендер inline keyboard | MEDIUM | При каждом взаимодействии бот отправляет новое сообщение с меню вместо редактирования существующего — засоряет чат историей сообщений |
| Нет обработки таймаутов HTTP | HIGH | Если backend недоступен — обработчик бота зависает без ответа пользователю |

### 21.2 MiniApp UX внутри Telegram

**Что работает хорошо:**
- Обнаружение платформы и тема синхронизируются с темой Telegram (`themeStore.initTheme()`)
- `useBackButton` корректно регистрирует Telegram BackButton API
- Haptic feedback используется через `telegramStore.hapticFeedback()` — нативное ощущение
- `showAlert()` использует нативный Telegram alert вместо кастомного модала — правильно

**Проблемы:**
1. **MainButton не используется** — Telegram MainButton (нативная фиолетовая кнопка внизу) не задействована. Вместо неё реализованы кастомные кнопки с `position: sticky; bottom: 0` — визуально схоже, но пользователи Telegram ожидают нативного поведения MainButton для основных CTA.
2. **Нет inline-клавиатуры для быстрых действий** — при переходе из бота в MiniApp нет "возврата" в бот с контекстом. Если пользователь закрывает MiniApp, он видит только последнее сообщение бота без актуального контекста.
3. **`INIT_DATA_MAX_AGE_SECONDS = 5 * 60`** — клиентская проверка вызывает принудительное закрытие сессии через 5 минут, хотя сервер принимает initData до 24 часов. Пользователь видит ошибку без понятного объяснения и вынужден перезапускать MiniApp.
4. **Нет `viewport` lock** — при открытии клавиатуры (textarea в аттестации) MiniApp может некорректно изменять размер viewport в некоторых версиях Telegram.
5. **Оверлей онбординга** — модальный оверлей поверх ещё не загруженного контента создаёт ощущение "заблокированного" экрана. В Telegram-парадигме лучше показывать онбординг как экран с навигацией back.

### 21.3 Взаимодействие бот → MiniApp → бот

Текущий флоу:
```
Бот: /start → inline button "Открыть приложение" → MiniApp открывается → пользователь учится → закрывает
```

После закрытия MiniApp пользователь возвращается в чат с ботом. Бот не знает, что произошло в MiniApp. Нет:
- Сообщения о завершении курса/аттестации через бота
- Напоминания о незаконченных курсах через scheduled messages
- Мотивационных сообщений по итогам сессии

**Рекомендация:** Реализовать event-driven уведомления через бот: когда пользователь завершает аттестацию — бот присылает congratulation message с результатом.

### 21.4 Notification strategy

Текущее состояние: `notificationScheduler.js` запускает уведомления по расписанию, но нет Telegram push-уведомлений для событий реального времени (назначение курса, результат аттестации, получение бейджа).

Пользователь узнаёт о событиях только при следующем открытии MiniApp. Dashboard показывает кнопку Bell, которая ни к чему не подключена — **dead UI element, создающий ложные ожидания**.

## 22. АНАЛИЗ UX ПЛАТФОРМЫ MAX

### 22.1 Состояние поддержки MAX

Платформа MAX поддержана на уровне аутентификации (`maxAuthService.js`). Frontend использует общий код с Telegram через платформенную абстракцию.

**Что работает:**
- `getClientPlatform()` определяет платформу
- Тема синхронизируется аналогично Telegram
- `setTelegramRuntimeState()` (переиспользован для MAX) — функция работает

**UX-проблемы специфичные для MAX:**

| Проблема | Серьёзность | Описание |
|---|---|---|
| `setTelegramRuntimeState` для MAX-платформы | MEDIUM | Функция с именем "telegram" вызывается для MAX — путаница при поддержке |
| Нет MAX-специфичных UX-адаптаций | HIGH | MAX имеет отличный от Telegram UX API. Не используются нативные MAX-компоненты (если поддерживаются), нет проверки MAX-версии |
| Feature parity не задокументирован | MEDIUM | Непонятно, какие функции работают одинаково в MAX и Telegram, какие — только в одной платформе |
| Нет тестирования UX на MAX | HIGH | Нет E2E тестов для MAX-флоу — рассинхронизация может быть обнаружена только в production |
| Кросс-платформенная сессия | MEDIUM | Один refresh_token для обеих платформ без разграничения — при смене платформы могут быть конфликты сессий |

### 22.2 Дизайн для MAX vs Telegram

MAX и Telegram имеют разные визуальные парадигмы. В текущей реализации один и тот же CSS применяется к обеим платформам через класс `platform-mobile`/`platform-desktop`. Нет `platform-max` класса для специфичных стилей.

**Риск:** При изменении MAX API или добавлении MAX-native компонентов потребуется рефакторинг без чёткой точки кастомизации.

## 23. АНАЛИЗ ВИЗУАЛЬНОГО ДИЗАЙНА

### 23.1 Цветовая система

Реализована через CSS custom properties — профессиональный подход. Токены:
- `--accent: #6355f5` (light) / `--accent: #7b6fff` (dark) — фиолетовый акцент
- `--bg-primary`, `--bg-secondary`, `--text-primary`, `--text-secondary` — базовые токены
- Semantic tokens: `--success`, `--error`, `--warning`

**Проблемы:**
- Нет токена `--accent-hover` — состояния hover/active определяются в компонентах через opacity или hardcoded цвет
- Нет border-radius токена (используется `--skeleton-radius: 12px`, но не как общий токен)
- `--card-shadow` меняется между темами (light: `rgba(0,0,0,0.08)`, dark: `rgba(0,0,0,0.5)`), но нет токена для elevation-уровней
- Токены для overlay, backdrop, scrim — отсутствуют

### 23.2 Типографика

Шрифт: `Montserrat` как основной, fallback на системные шрифты. Типографическая шкала задана классами:
- `.title-large` (28px/700), `.title-medium` (22px), `.body-medium`, `.body-small`

**Проблемы:**
- Классы типографики (`.title-large`) смешиваются с utility-классами TailwindCSS в admin-части — нет единой системы
- Отсутствует `.title-small` в шкале, что вынуждает разработчиков использовать произвольные `font-size`
- Line-height задан глобально (`1.5`), но не масштабируется для заголовков и body отдельно
- Отсутствует `letter-spacing` токен — особенно важен для кнопок и caption текста

### 23.3 Консистентность компонентов

**MiniApp:** Компонентная система присутствует (SkeletonCard, BottomNavigation, Preloader), но:
- Кнопки реализованы через CSS-классы (`.btn`, `.btn-primary`, `.continue-btn`, `.task-item`, `.btn-final`) — 5+ вариантов без системного подхода
- `app-panel`, `app-grid-3`, `app-list-stack` — utility-классы компоновки не задокументированы
- Нет состояния `focus-visible` для кнопок — важно для доступности

**Admin Panel:** Переход к shadcn-vue задокументирован в дизайн-системе, но не завершён. Смешаны:
- TailwindCSS utility classes
- Кастомные CSS-компоненты
- shadcn-vue (планируется)

### 23.4 Иконки

Lucide выбрана как единый icon set — правильное решение. Используется `lucide-vue-next`. Найдены: Bell, BookOpen, ShieldCheck, Check, Lock, GraduationCap, и другие. Единообразие соблюдается.

### 23.5 Визуальная иерархия

**MiniApp Dashboard:**
- Приветственный заголовок (h1) — хорошо
- Кнопка уведомлений (26px, рядом с заголовком) — конкурирует по размеру с заголовком
- Секции с `section-title` — второй уровень иерархии — корректно

## 24. АНАЛИЗ МОБИЛЬНОГО UX

### 24.1 Thumb-zone и эргономика

MiniApp спроектирован для мобильного использования:
- Bottom navigation (5 иконок) — в нижней части экрана, большой thumb-zone ✅
- Sticky footer CTA ("Продолжить") — доступна большим пальцем ✅
- `padding-bottom: 76px` для body — предотвращает перекрытие контента bottom nav ✅

**Проблемы:**
- Кнопка уведомлений (Bell) на Dashboard — верхний правый угол, **мёртвая зона** большого пальца на больших смартфонах (> 6 дюймов)
- В CourseDetailsView кнопка секции `section-card` — полная ширина, хорошо. Но иконка замка в правом углу — маленькая touch target (18px)
- В TopicView список подтем — кнопки без указания min-height, могут быть < 44px (нарушение Apple HIG и Material)

### 24.2 Размеры touch-targets

Telegram HIG рекомендует минимум 44×44px для интерактивных элементов. Проверено по коду:
- `section-card` — нет явного `min-height`, контент определяет высоту
- `stat-card` — информационный, не интерактивный ✅
- `matching-chip` кнопки — нет `min-height`, только padding ⚠️
- `section-lock-icon` (18px) — кликабельна в составе кнопки, но визуально маленькая

### 24.3 Scroll и контент

- `overflow-y: scroll` или `auto` не задан глобально — возможны конфликты
- В SubtopicView rich text контент может содержать изображения без `max-width: 100%` — возможен горизонтальный скролл
- Matching question на мобильном: при 5+ парах список chip-кнопок даёт длинный вертикальный скролл без "закрепления" левой части

### 24.4 Адаптивность Admin Panel

Admin Panel использует TailwindCSS. По коду не выявлено явных `sm:`, `md:`, `lg:` breakpoint-классов в ключевых компонентах, что может означать отсутствие адаптивности для планшетов.

**Проблема:** Администраторы могут работать с iPad — отсутствие `md:`-варианта сайдбара (collapsed/expanded) и адаптивных таблиц создаёт сложности.

### 24.5 Keyboard interaction

В AssessmentProcessView textarea для text-вопросов — при открытии виртуальной клавиатуры viewport уменьшается. Если sticky footer не учитывает `env(keyboard-inset-height)`, кнопка "Далее" окажется под клавиатурой.

## 25. АНАЛИЗ ДОСТУПНОСТИ

### 25.1 Семантическая структура

**Что реализовано корректно:**
- `role="alert"` на `.error-banner` в RegistrationView ✅
- `aria-label="Уведомления"` на кнопке Bell ✅
- `alt=""` (декоративное) на изображении курса в Dashboard ✅
- `<article>` для карточек курсов, `<section>` для групп — семантика используется ✅
- `<h1>`, `<h2>` иерархия в большинстве view соблюдена ✅

**Проблемы:**
- `id` для `<textarea>` в аттестации (`text-answer-${currentQuestion.id}`) с соответствующим `<label for>` ✅ — корректно
- Кнопки с только иконкой (например, matching clear: "Сбросить выбор") — текст есть, но стилистически выглядит как icon-only ⚠️
- `v-for` списки без `aria-current` на активном элементе
- Overlay онбординга не фокусирует кнопку при появлении — `focus-trap` отсутствует

### 25.2 Контраст

**Светлая тема:**
- `--text-primary: #111827` на `--bg-primary: #ffffff` — соотношение ~18:1 ✅
- `--text-secondary: #6b7280` на `--bg-primary: #ffffff` — соотношение ~4.6:1 (WCAG AA проходит для текста 18px+, fail для меньшего) ⚠️
- `--accent: #6355f5` как текст на белом фоне — ~5.5:1 ✅

**Тёмная тема:**
- `--text-secondary: #7a7a90` на `--bg-primary: #111118` — соотношение ~3.8:1 ⚠️ (ниже WCAG AA 4.5:1 для обычного текста)
- `--skeleton-bg: #2a2d39` — используется только для skeleton, не для текста ✅

### 25.3 Keyboard навигация

- Tab-порядок в MiniApp: не проверен явно, но компоненты используют кнопки и ссылки — нативный tab-order должен работать
- Admin Panel: TailwindCSS не добавляет автоматических focus-стилей — необходимо проверить наличие `focus-visible` стилей
- В CourseEditorView множество интерактивных элементов (drag handles, кнопки секций, кнопки тем) — без явного `tabindex` управления

### 25.4 Screen reader совместимость

- `aria-hidden="true"` на декоративных изображениях ✅
- Прогресс-бар аттестации: `<div class="progress-fill" :style="...">` — нет `role="progressbar"` с `aria-valuenow`/`aria-valuemax` ⚠️
- Статистика на Dashboard: `<p class="stat-value">{{ dashboardStats.courses }}</p>` — нет контекста для screen reader, только число без подписи в одном элементе

## 26. ОЦЕНКА КАЧЕСТВА ОБРАЗОВАТЕЛЬНОГО ОПЫТА

### 26.1 Качество флоу обучения

**Структура LMS:**
Курс → Секция (тема) → Подтема → Контент (rich text + media)

Линейная прогрессия с блокировкой (locked sections) — правильная педагогическая модель. Таймер чтения обеспечивает минимальное время взаимодействия с материалом.

**Проблемы образовательного опыта:**

1. **Таймер чтения невидим** — пользователь не знает о минимальном времени чтения, пока кнопка "Завершить" неактивна. Нет индикатора "подождите ещё X секунд". Это воспринимается как баг, а не фича.
2. **Нет форматирования прогресса внутри подтемы** — при длинном rich text контенте нет индикации сколько осталось до конца (нет scroll progress indicator).
3. **Контент только rich text** — нет нативной поддержки видео-плеера, аудио, интерактивных элементов. Iframe через Tiptap — единственный способ встроить внешний контент, что ограничивает педагогические возможности.

### 26.2 Геймификация и мотивация

**Реализовано:**
- Баллы, уровни, бейджи — система есть
- Лидерборд (отдельный экран) — стимулирует соревновательность
- Достижения в профиле — видимость прогресса

**Слабые стороны:**
1. **Геймификация изолирована** — лидерборд и достижения живут в отдельных экранах, не интегрированы в процесс обучения. Студент не видит "ты получил 50 баллов" сразу после завершения темы — только при следующем визите в профиль.
2. **Нет streak-механики** — нет поощрения ежедневного входа и обучения.
3. **Бейджи без контекста** — нет объяснения за что дан конкретный бейдж в момент получения.
4. **Лидерборд без фильтрации** — нет возможности увидеть рейтинг по своему отделу/филиалу.

### 26.3 Возвращаемость пользователей

Механизмов возврата критически мало:
- Нет streak
- Нет email/push уведомлений о назначенных курсах
- Нет напоминаний через бот о незаконченных курсах
- "Ближайшие задачи" на dashboard — единственный ориентир, но нет дедлайнов

### 26.4 Feedback-система

- После завершения аттестации: `AssessmentResultsView` с результатом — есть ✅
- После завершения темы: переход к следующей теме или CourseStatusPage ✅
- После завершения курса: `CourseStatusPageView` с 6 сценариями ✅
- **Нет**: анимации успеха ("confetti"), звуковой/тактильной награды за прохождение аттестации, inline сообщения "Отличная работа!"

## 27. ОЦЕНКА UX ПАНЕЛИ АДМИНИСТРАТОРА

### 27.1 Dashboard администратора

Основной дашборд администратора предположительно включает Chart.js графики аналитики (задокументировано в аудите секции 12.2).

**Проблемы:**
1. **Нет overview метрик** на первом экране после входа — нужно идти в конкретный раздел для получения данных
2. **Нет алертов о проблемах** — если студент застрял на курсе, если попытка аттестации провалена много раз — администратор не получает уведомления
3. **Нет quick actions** — создание курса, назначение пользователя, отправка уведомления требуют навигации в соответствующий раздел

### 27.2 Управление курсами

CourseEditorView — 4-шаговый wizard. Проблемы выявлены в секции 20.4. Дополнительно:
- **Нет версионирования курсов** в UI — нельзя откатить к предыдущей версии контента
- **Нет предпросмотра** — администратор не может посмотреть курс глазами студента
- **Шаг "Структура" перегружен** — одновременно содержит: список секций с drag-handle, список тем, форму создания секции, форму создания темы, редактор материала. Это 5 функциональных зон на одном экране.

### 27.3 Управление пользователями

**Выявлено:**
- Нет bulk-действий (назначить курс всему отделу)
- Нет фильтрации пользователей по прогрессу (кто не начинал курс, кто застрял)
- Нет экспорта данных о прогрессе в CSV/Excel для отчётности HR

### 27.4 Аналитика

Chart.js графики подключены. Но:
- **Аналитика не real-time** — данные обновляются при загрузке страницы, нет auto-refresh
- **Нет drill-down** — нельзя нажать на секцию графика и увидеть детали
- **Нет сравнения периодов** — нет сопоставления "эта неделя vs прошлая"

### 27.5 Скорость workflow

|---|---|---|
| Создать курс (заполнить основную информацию) | ~8–12 | Приемлемо |
| Добавить секцию и тему с контентом | ~15–20 | Много |
| Назначить курс пользователю | ~6–8 | Приемлемо |
| Просмотреть прогресс конкретного студента | ~4–6 | Норм |
| Отправить уведомление | N/A | Функция не задокументирована в UI |

## 28. АНАЛИЗ КОНВЕРСИИ И УДЕРЖАНИЯ

### 28.1 Онбординг-конверсия

Воронка нового пользователя:
```
Получить ссылку-приглашение (внешний канал) 
→ Открыть MiniApp 
→ RegistrationView (нажать "Подтвердить номер") 
→ Telegram requestContact (системный диалог) 
→ Backend auth 
→ Онбординг-оверлей (нажать "Понятно, начать") 
→ Dashboard
```

**Drop-off точки:**
1. **Шаг "requestContact"** — часть пользователей откажется от предоставления номера телефона. Нет fallback пути (ввести вручную).
2. **"Номер не найден"** — пользователь получает ошибку и инструкцию запросить приглашение. Нет способа сделать это прямо из MiniApp.
3. **Онбординг-оверлей** — хотя и минимален (одна кнопка), задерживает доступ к контенту.

### 28.2 Механизмы удержания

| Механизм | Статус | Оценка |
|---|---|---|
| Геймификация (баллы/бейджи) | ✅ Реализована | Средняя — не интегрирована в поток |
| Лидерборд | ✅ Реализован | Слабая — нет социального контекста (друзья, отдел) |
| Уведомления через бот | ⚠️ Частично | Слабая — нет real-time push |
| Streak / ежедневный вход | ❌ Отсутствует | — |
| Email-уведомления | ❌ Отсутствует | — |
| Напоминания о незаконченных курсах | ❌ Отсутствует | — |
| Дедлайны курсов | ❌ Не реализованы в UX | — |

### 28.3 Риски churn

1. **Нет дедлайнов** — пользователи без внешнего давления откладывают обучение на потом
2. **Нет напоминаний** — без push/бот-напоминаний пользователи забывают о незаконченных курсах
3. **5-минутный timeout сессии** — создаёт разочарование при активном использовании
4. **Нет прогресс-мотиватора** — нет "ты прошёл 60%! ещё 2 темы до сертификата" на промежуточных экранах

## 29. ОЦЕНКА КОМПОНЕНТНОЙ СИСТЕМЫ

### 29.1 MiniApp — состояние компонентной системы

**Реализованные компоненты:**
- `BottomNavigation` — глобальная навигация
- `Preloader` — загрузочное состояние
- `SkeletonCard`, `SkeletonPageHeader`, `SkeletonBlock`, `SkeletonList` — skeleton-система
- Компоненты курса и аттестации

**Проблемы:**
- **Кнопки не выделены в компонент** — 5+ CSS-классов для кнопок (`.btn`, `.continue-btn`, `.btn-final`, `.task-item`) без единого `<Button>` компонента с вариантами
- **Нет компонента `<Card>`** — карточки (`app-panel`, `stat-card`, `continue-card`) реализованы через CSS-классы, не Vue-компоненты
- **Нет компонента `<ProgressBar>`** — прогресс-бар реализован 3 раза в разных компонентах с разным CSS
- **Нет Toast-компонента** — ошибки показываются через Telegram native alert (`showAlert`), нет inline toast/snackbar для информационных сообщений

### 29.2 Admin Panel — состояние компонентной системы

`docs/design/system.md` фиксирует цель — переход на shadcn-vue паттерн. В `admin-frontend/src/components/ui/` должны находиться системные компоненты.

**По структуре `components/` выявлено:**
- `charts/`, `courses/`, `layout/`, `ui/` — папки организованы правильно
- `PermissionMatrixComponent.vue` — standalone компонент в корне `components/` без namespace — нарушение структуры
- Использование `vue-sonner` (Toaster) для Toast — стандартное решение ✅

### 29.3 Токены и переменные

**MiniApp:** CSS custom properties — хорошо, но не все свойства токенизированы (border-radius, z-index слои, animation duration).

**Admin Panel:** TailwindCSS предоставляет свою систему токенов. При миксе с CSS variables возникает риск дублирования (цвет акцента определён и в Tailwind config, и в CSS vars).

### 29.4 Масштабируемость UI

При текущем состоянии добавление нового экрана требует:
- Написания кастомных CSS классов (нет компонентной базы)
- Ручного соблюдения дизайн-паттернов без design system enforcement
- Дублирования кода прогресс-баров, skeleton, кнопок

**Оценка масштабируемости UI: низкая** — каждый новый экран увеличивает технический долг UI.

## 32. БЫСТРЫЕ UX-ПОБЕДЫ

Минимальные усилия — максимальный эффект на UX и удержание:

1. **Убрать кнопку Bell или подключить** *(30 мин)* — устраняет dead UI, возвращает доверие к интерфейсу
2. **Согласовать INIT_DATA таймаут** *(15 мин)* — устраняет принудительный logout, снижает drop-off
3. **Countdown таймера чтения** *(2 ч)* — "Ещё 15 сек." под кнопкой Завершить — устраняет ощущение бага
4. **Dashboard CTA → SubtopicView** *(4 ч)* — убирает 3 лишних клика при ежедневном использовании
5. **`role="progressbar"` на прогресс-барах** *(1 ч)* — WCAG compliance без дизайн-изменений
6. **`focus-visible` глобально** *(30 мин)* — keyboard accessibility одной строкой CSS
7. **`min-height: 44px` на интерактивных элементах** *(2 ч)* — устранение mis-tap на мобильных
8. **Текст контраст тёмной темы** *(30 мин)* — `--text-secondary` с 3.8:1 до 4.5:1 одним токеном
9. **Перевод бота на webhook** *(3 ч)* — мгновенный ответ вместо polling-задержки
10. **`aria-label` на section-lock-icon** *(30 мин)* — "Раздел заблокирован" для screen reader
11. **Бот-уведомление о результате аттестации** *(8 ч)* — первый шаг retention-механики через существующий бот

## 33. ФИНАЛЬНЫЙ UI/UX ВЕРДИКТ

### Итоговые оценки

| Критерий | Оценка | Комментарий |
|---|---|---|
| **Общий продуктовый опыт** | **6.2 / 10** | Рабочий LMS, но с ощутимыми фрикциями в ключевых флоу |
| **UX-зрелость** | **Intermediate** | Базовые паттерны реализованы, edge cases и retention — нет |
| **Масштабируемость UI** | **Низкая** | Отсутствие компонентной системы увеличивает долг с каждым экраном |
| **Telegram UX-качество** | **5.5 / 10** | MiniApp работает, бот без FSM и webhook, dead UI elements |
| **MAX UX-качество** | **5.0 / 10** | Auth поддержан, UI без платформо-специфичных адаптаций |
| **Мобильный опыт** | **6.5 / 10** | Mobile-first CSS, skeleton, bottom nav — хорошо. Touch targets и deep nav — плохо |
| **Доступность** | **4.5 / 10** | Семантика частично, aria-атрибуты прогресса отсутствуют, контраст в dark mode ниже WCAG |
| **Дизайн-система** | **3.5 / 10** | CSS токены есть, компонентная система не завершена, shadcn-vue в плане |
| **Конкурентоспособность** | **Средняя** | По функциональности — достаточно для корпоративного LMS; по полировке UX уступает коммерческим аналогам |

### Долгосрочные UX-риски

1. **Retention crisis** — без push-уведомлений и streak-механики пользователи перестают открывать MiniApp без внешнего принуждения (административного давления). Engagement полностью зависит от HR-напоминаний, а не от продукта.
2. **UI-технический долг** — без дизайн-системы каждый новый экран добавляет несогласованные CSS-классы. Через 12 месяцев поддержка UI станет дорогостоящей.
3. **Telegram-специфичный риск** — опора на Telegram WebApp API без fallback создаёт риск при изменениях API со стороны Telegram (прецеденты были с Bot API).
4. **MAX-платформа** — без дополнительного UX-тестирования на MAX есть риск незамеченных регрессий при обновлении платформы.

### Ключевые сильные стороны UX

1. **Тёмная тема и CSS-токены** — профессиональная реализация, мгновенное переключение без артефактов
2. **Skeleton-система** — все экраны загружаются без прыжков контента, отличный perceived performance
3. **Status pages** — 6 контекстуальных сценариев покрывают edge cases курсового прогресса
4. **Haptic feedback** — нативное тактильное ощущение в ключевых моментах взаимодействия
5. **Адаптивный matching UI** — разные режимы для mobile/desktop — продуманное решение
6. **Wizard создания курса** — 4-шаговый stepper со структурированным подходом

### Рекомендация по инвестиционному приоритету UX

**Краткосрочно (0–1 мес.):** Быстрые победы (секция 32) — ROI максимальный при минимальных затратах. Особый приоритет: INIT_DATA таймаут, таймер чтения, Dashboard CTA.

**Среднесрочно (1–3 мес.):** Push-уведомления через бот и inline геймификация — наибольшее влияние на retention. Завершение shadcn-vue в Admin.

**Долгосрочно (3–6 мес.):** Дизайн-система MiniApp (компонентная библиотека), streak-механика, offline graceful degradation.

**Без этих инвестиций** продукт останется функционально работающим LMS с административным forced-use, но без органической вовлечённости пользователей — что критично для корпоративной платформы, где completion rate курсов напрямую влияет на бизнес-метрики.
