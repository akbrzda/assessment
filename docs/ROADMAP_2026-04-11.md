# Roadmap проекта assessment

Дата: 2026-04-11  
Основание: `docs/features.md` + `docs/AUDIT_REPORT.md` + актуализация `docs/AUDIT_CONCLUSION_2026-04-11.md`

## Принцип приоритизации (по вашему запросу)

1. Сначала **Курсы** (как главный продуктовый приоритет).
2. Затем **доработка текущего функционала** (включая все найденные проблемы из аудита).
3. Потом **новый функционал с нуля**.

## Приоритет 1 — Курсы (главный)

## Цель

Собрать полноценную учебную траекторию: курс -> темы -> итоговая аттестация курса.

## Объем работ

1. Модель данных курсов:
   - `courses`;
   - `course_topics`;
   - связи курса с существующими аттестациями/темами;
   - `course_attempts`, `course_progress`.
2. Админ-панель:
   - CRUD курсов;
   - конструктор тем курса;
   - назначение курсов по филиалам/должностям/пользователям.
3. MiniApp:
   - список курсов;
   - прохождение тем по порядку;
   - финальная аттестация курса.
4. Аналитика курса:
   - % прохождения;
   - средний балл;
   - затраченное время;
   - количество попыток.

## Критерии завершения

- Курс проходит полный цикл в admin и miniapp.
- Видна отдельная статистика: тема / финал / курс.
- Назначение и доступы работают с учетом роли и филиала.

## Приоритет 2 — Доработка текущего функционала (включая аудит)

## 2.1 Безопасность и доступы (из аудита)

1. Убрать fallback JWT-секретов, оставить только env + fail-fast.
2. Зафиксировать строгий `ALLOWED_ORIGINS` для production.
3. Добавить CSRF-защиту для refresh cookie-flow.
4. Добавить rate-limit/anti-bruteforce на admin login.
5. Убрать хранение access token в `localStorage` (перевести в memory + cookie flow).
6. Проверка `auth_date` для Telegram initData (анти-replay окно).
7. Довести изоляцию менеджера по филиалу во всех связанных сценариях.
8. Подключить `helmet`/CSP/security headers.

## 2.2 Исправления логики текущих модулей

1. Аттестации:
   - исправить логику `matching`;
   - исправить SQL с неверной таблицей для неполных попыток;
   - корректно реализовать возобновление попытки и фиксированный порядок вопросов/вариантов;
   - автозавершение по таймеру без регрессий.
2. Аудит и логи:
   - реализовать реальную запись аудита (БД/хранилище);
   - отправка событий в Telegram-группу по thread.
3. Инвайты и регистрация:
   - закрыть self-registration без приглашения;
   - сделать атомарное погашение инвайта (без гонок).
4. Теория:
   - расширить текущую реализацию блоков (`iframe`, `image`, `audio`) как штатные типы контента.
5. Геймификация:
   - ручная выдача бейджей в карточке пользователя;
   - ручная корректировка очков/уровней с обязательным аудитом.
6. Производительность и качество запросов:
   - убрать `SELECT *` в прикладном коде;
   - добавить пагинацию на тяжелых списках;
   - user-scoped cache.
7. Миграции:
   - таблица истории миграций (`schema_migrations`);
   - строгая нумерация;
   - устойчивый runner (без хрупкого split SQL).

## 2.3 Операционная готовность

1. Включить quality gates в CI:
   - lint;
   - tests;
   - build.
2. Убрать лишние debug-логи из production-кода.
3. Привести матрицу модулей доступа backend/frontend к единому источнику.

## Критерии завершения приоритета 2

- Все критичные/высокие пункты из аудита закрыты.
- Текущий функционал работает стабильно без критичных дефектов.
- Деплой блокируется при провале quality gates.

## Приоритет 3 — Новый функционал с нуля

## 3.1 Уведомления и бот-платформа

1. Event pipeline уведомлений:
   - назначены аттестации;
   - открылись аттестации;
   - результаты;
   - бейджи/уровни.
2. Полноценный бот:
   - кнопки deep-link в miniapp;
   - FAQ;
   - onboarding;
   - уведомления по новым материалам БЗ.

## 3.2 Мультиязычность

1. i18n для интерфейсов admin/miniapp.
2. Мультиязычный контент аттестаций и теории.
3. Выдача контента по языку пользователя.

## 3.3 Нечеткие эталонные ответы

1. Триггер-слова.
2. Нормализация текста.
3. Fuzzy matching и правила оценивания.

## 3.4 GiftShop

1. Каталог товаров/настроек.
2. Списание и резерв баллов.
3. Заказы, статусы, выдача.
4. Аудит всех финансовых операций баллов.

## 3.5 Продвинутая отчетность

1. Плановая выгрузка в Excel/Google Sheets.
2. Конструктор отчетов (сохраненные шаблоны).
3. OLAP-подобные срезы по филиалам/городам/ролям.

## Критерии завершения приоритета 3

- Новый функционал работает end-to-end.
- Есть эксплуатационная поддержка (логи, мониторинг, аудит).
- Нет критичных отклонений по безопасности.

## Сводный порядок запуска

1. Приоритет 1: Курсы.
2. Приоритет 2: Доработка текущего функционала + все пункты аудита.
3. Приоритет 3: Новый функционал с нуля.
