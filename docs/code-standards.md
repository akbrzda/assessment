# 📘 Engineering Standard — Code, Architecture & Maintainability

---

## 0. Уровни строгости

- **MUST** — обязательно. Нарушение блокирует PR
- **SHOULD** — рекомендуется. Требуется аргумент
- **FORBIDDEN** — запрещено

---

## 1. Цель

Документ задаёт единые стандарты разработки для всех сервисов.

Цели:

- код читается без объяснений;
- архитектура предсказуема;
- модули изолированы;
- изменения локализованы;
- проект масштабируется без переписывания ядра.

---

## 2. Общие принципы

### 2.1 Один файл — одна ответственность

- файл **MUST** иметь одну причину для изменения
- функция **MUST** работать на одном уровне абстракции

---

### 2.2 Предсказуемость важнее магии

- код **MUST** быть очевидным
- сложные абстракции **SHOULD** избегаться
- если механизм нельзя объяснить за 5–10 минут — он **FORBIDDEN**

---

### 2.3 Независимость бизнес-логики

Бизнес-логика **MUST NOT** зависеть от:

- Express
- Telegram API
- Vue/React
- Axios
- конкретной БД

---

## 3. Архитектура модулей

### 3.1 Общая структура

```text
modules/<domain>/
  routes.js
  controller.js
  service.js
  repository.js
  validators.js
  mapper.js
  index.js
```

---

### 3.2 Границы модулей

- модуль **MUST** иметь публичный API (`index.js`)
- доступ к модулю — только через него

**FORBIDDEN:**

```js
import { something } from "modules/user/service.js";
```

---

### 3.3 Зависимости

Разрешено:

```
routes → controller → service → repository
```

**FORBIDDEN:**

- обратные зависимости
- доступ к внутренностям модуля
- cross-module хаотичные импорты

---

### 3.4 Low Coupling / High Cohesion

- модуль **MUST** быть сфокусирован
- зависимости **MUST** быть минимальны

---

## 4. Признаки плохого модуля

Модуль считается перегруженным, если:

- > 5–7 импортов из разных слоёв
- знает про API + router + storage + UI
- изменяется по разным причинам
- содержит бизнес-логику и инфраструктуру

→ MUST split

---

## 5. Контракты данных

### 5.1 Вход

- данные **MUST** валидироваться
- валидация происходит до service

---

### 5.2 Выход

- данные **MUST** нормализоваться (DTO / mapper)
- формат ответа **MUST** быть стабильным

---

## 6. Ограничение абстракций

Абстракция допустима только если:

- есть ≥2 кейса
- снижает сложность

Иначе — **FORBIDDEN**

---

## 7. Backend архитектура

### 7.1 Слои

| Слой       | Ответственность  |
| ---------- | ---------------- |
| routes     | маршруты         |
| controller | HTTP             |
| service    | бизнес-логика    |
| repository | работа с данными |

---

### 7.2 Ограничения

**FORBIDDEN:**

- SQL в routes
- req/res в service
- HTTP-ошибки в repository

---

## 8. Frontend архитектура

### 8.1 Структура

```text
src/
  router/
  stores/
  services/
  api/
  utils/
  views/
  components/
```

---

### 8.2 Store

Store:

**MUST:**

- управлять состоянием

**FORBIDDEN:**

- быть вторым backend
- управлять всем сразу (API + router + websocket + storage)

---

### 8.3 Router

- декларативный
- guards разделены

**FORBIDDEN:**

- бизнес-логика
- websocket / analytics логика

---

### 8.4 Telegram / Mini App

- Telegram код изолирован
- нет глобальных window.\_\_\*

---

## 9. HTTP и Session

- refresh логика **MUST** быть централизована
- один источник правды

**FORBIDDEN:**

- дублирование refresh логики

---

## 10. Импорты

Порядок:

1. внешние пакеты
2. внутренние алиасы
3. относительные

---

## 11. Именование

**MUST:**

- отражать смысл

**FORBIDDEN:**

- data, tmp, obj, res2

---

## 12. Размер кода

Файлы:

- <150 — норма
- 150–250 — допустимо
- > 250 — аргумент
- > 350 — MUST split

Функции:

- <25 строк — норма
- > 50 — разбивать

---

## 13. Обработка ошибок

**MUST:**

- нет пустых catch
- ошибки разделены
- ошибки логируются

---

## 14. Логирование

**MUST:**

- единый logger
- без чувствительных данных

---

## 15. Env и конфиги

**MUST:**

- `.env.example`
- валидация env

**FORBIDDEN:**

- fallback secrets

---

## 16. Тесты

Минимум:

- auth
- permissions
- критичные flows

Приоритет:

1. unit
2. integration
3. e2e

---

## 17. Анти-паттерны

**FORBIDDEN:**

- god-file
- god-store
- utils-помойка
- дублирование логики
- сложные абстракции

---

## 18. Добавление фичи

1. определить домен
2. создать модуль
3. разделить слои
4. добавить validation
5. добавить mapper
6. проверить зависимости
7. добавить тесты

---

## 19. Чек-лист разработчика

- [ ] правильный слой
- [ ] нет смешения логики
- [ ] зависимости корректны
- [ ] код читаемый
- [ ] нет дублирования
- [ ] есть validation
- [ ] есть обработка ошибок

---

## 20. Чек-лист ревьюера

- [ ] соблюдены границы модулей
- [ ] нет god-files
- [ ] нет дублирования
- [ ] зависимости корректны
- [ ] код расширяем

---

## 21. Красные флаги

PR блокируется если:

- god-file
- импорт внутренностей модуля
- бизнес-логика вне service
- дублирование auth
- сложная абстракция без причины

---

## 22. Масштабируемость

Код считается масштабируемым если:

- изменение локально
- легко добавить фичу
- можно заменить зависимость

---

## 23. Definition of Done

- архитектура соблюдена
- код читаемый
- зависимости корректны
- ошибки обработаны
- тесты есть

---

## 24. Действия в репозитории

1. прекратить рост legacy
2. писать только через modules
3. централизовать auth
4. убрать глобальные состояния
5. внедрить lint/test baseline
6. зафиксировать CI проверки
