# Отчёт по доработке бота аттестации

> **Версия:** 1.0  
> **Дата:** 2026-05-05  
> **Статус:** Готов к согласованию  
> **Назначение:** Постановка задач для разработки, согласование с заказчиком, проектирование backend/UX

---

## 1. Краткое резюме

Проект представляет собой экосистему аттестации сотрудников: Telegram-бот + MiniApp (WebApp) + Admin Panel + Backend на Node.js/MySQL. Текущий Telegram-бот выполняет единственную функцию — открытие MiniApp через кнопку при команде `/start`. Уведомления, онбординг, напоминания и сертификаты не реализованы.

Цель доработки — превратить бот из «точки входа» в полноценный канал взаимодействия с пользователем: информировать о новых курсах, напоминать о незавершённых, сообщать о результатах и выдавать подтверждение прохождения.

**Ключевые выводы:**

- Фундамент готов: все сущности (пользователи, курсы, попытки, прогресс, бейджи, уровни) присутствуют в БД.
- `onboarding_completed_at` в таблице `users` уже предусмотрено — поле ждёт логики.
- Система уведомлений отсутствует полностью: нет таблиц, нет сервисов, нет cron.
- Сертификаты не предусмотрены в схеме БД — требуется проектирование с нуля.
- Бот не хранит `chat_id` пользователей для проактивных отправок — критический пробел.

## 2. Цели доработки

| #   | Цель                              | Бизнес-эффект                                          |
| --- | --------------------------------- | ------------------------------------------------------ |
| 1   | Онбординг нового сотрудника       | Снизить отток при первом входе, ускорить вовлечение    |

| 3   | Напоминание о незавершённом курсе | Повысить completion rate до дедлайна                   |
| 4   | Уведомление о результате          | Закрыть петлю обратной связи, мотивировать             |
| 5   | Выдача сертификата                | Подтвердить квалификацию, создать ценность прохождения |

## 3. Предположения по текущей логике проекта

**Подтверждено анализом кода:**

- Telegram-бот написан на **Telegraf** (Node.js), файл `bot/src/index.js`.
- При первом `/start` бот отображает приветствие с кнопкой «Открыть MiniApp» (Telegram WebApp).
- При `/start invite_CODE` обрабатывается реферальный код из таблицы `invitations`.
- Авторизация в MiniApp — через `Telegram.WebApp.initData` (HMAC-SHA256), не сессии бота.
- Таблица `users` содержит `onboarding_completed_at` (NULL = онбординг не завершён).
- Таблица `courses` содержит `status` (published/archived) и `close_at` (дедлайн).
- Таблица `assessment_attempts` фиксирует прогресс пользователя по тестам.
- Прогресс по курсам хранится в `course_user_progress` и `course_module_user_progress`.
- `telegram_id` пользователя доступен в БД — но `chat_id` бота (для отправки сообщений) не хранится отдельно.

**Допущение:** `telegram_id` и `chat_id` в Telegram совпадают для личных чатов (direct messages). Это стандартное поведение — `chat_id` личного чата равен `telegram_id` пользователя. Подтверждения из кода нет, но риск низкий.

**Пробелы:**

1. Нет таблицы `bot_notifications` (история отправок, статусы доставки).
2. Нет таблицы `certificates` (сертификаты).
3. Нет cron/планировщика для уведомлений.
4. Нет поля `notifications_enabled` в `users` (управление согласием на уведомления).
5. Нет привязки курса к пользователю (`course_assignments`) — курсы публикуются глобально или по доступности.

## 5. Онбординг в боте аттестации

### 5.1 Цель функции

Провести нового пользователя через первый запуск бота: объяснить, что такое система аттестации, познакомить с возможностями, побудить открыть первый курс. Зафиксировать факт завершения онбординга.

**Бизнес-ценность:** Снижение «брошенных» пользователей после первого входа. Без онбординга пользователь видит кнопку «Открыть приложение» — и не понимает, что делать дальше.

### 5.2 Пользовательский сценарий

```
Сценарий A: Новый пользователь по ссылке-приглашению
─────────────────────────────────────────────────────
1. Сотрудник получает от менеджера ссылку вида:
   https://t.me/theorica_bot?start=invite_XXXXXX

2. Переходит в Telegram → бот автоматически получает /start invite_XXXXXX

3. Backend проверяет код приглашения:
   - Валидный → профиль предзаполнен (ФИ, должность, филиал)
   - Невалидный → сообщение об ошибке (см. Edge cases)

4. Бот отправляет последовательность сообщений онбординга:
   Шаг 1 → Шаг 2 → Шаг 3 → Финальное сообщение с CTA

5. После взаимодействия с последним шагом:
   - Записывается onboarding_completed_at в таблицу users
   - Открывается MiniApp (или предлагается кнопка)

Сценарий B: Прямой /start без приглашения
───────────────────────────────────────────
1. Пользователь отправляет /start напрямую
2. Если users.telegram_id не найден → онбординг-приветствие без профиля
3. Предлагается ввести код приглашения или связаться с менеджером
4. Если users.telegram_id найден и onboarding_completed_at IS NULL → онбординг
5. Если onboarding_completed_at заполнен → стандартное меню бота
```

### 5.3 Логика сообщений

**Шаг 1 — Приветствие**

```
👋 Привет, [Имя]!

Я бот системы аттестации [Название компании].

Здесь ты будешь:
📚 Проходить обучающие курсы
✅ Сдавать тесты и аттестации
🏆 Получать баллы и бейджи за прогресс
📄 Получать сертификаты о прохождении

Давай я покажу, как это работает.
```

**Шаг 2 — Как устроена система**

```
📖 Как это работает:

1. Тебе назначаются курсы
2. Ты изучаешь материалы в удобном темпе
3. Проходишь тест в конце курса
4. Получаешь результат и сертификат

Все курсы доступны прямо здесь, в приложении внутри Telegram.
```

**Шаг 3 — Первый шаг**

```
🚀 Готов начать?

Нажми кнопку ниже, чтобы открыть приложение.
Там тебя уже ждут курсы!

[Открыть приложение] ← кнопка WebApp
```

### 5.4 Кнопки и действия

| Шаг       | Элемент                            | Действие                                           |
| --------- | ---------------------------------- | -------------------------------------------------- |
| Шаг 1     | Кнопка «Далее» или inline          | Переход к шагу 2                                   |
| Шаг 2     | Кнопка «Понятно»                   | Переход к шагу 3                                   |
| Шаг 3     | Кнопка WebApp «Открыть приложение» | Открыть MiniApp + записать onboarding_completed_at |
| Любой шаг | Кнопка «Пропустить»                | Перейти к финалу, записать onboarding_completed_at |

**Вариант реализации без шагов (упрощённый MVP):**
Одно приветственное сообщение + кнопка «Открыть приложение». Запись `onboarding_completed_at` при открытии MiniApp через отдельный API-эндпоинт.

### 5.5 Административная настройка

- Текст онбординга хранится в `system_settings` (существующая таблица).
- Ключи: `BOT_ONBOARDING_STEP_1`, `BOT_ONBOARDING_STEP_2`, `BOT_ONBOARDING_STEP_3`.
- Администратор может изменить текст через Admin Panel без деплоя.
- Флаг `BOT_ONBOARDING_ENABLED` (boolean) — включить/выключить онбординг.

### 5.6 Техническая реализация

```
bot/src/scenes/onboarding.js   ← сцена онбординга (Telegraf Scenes)
bot/src/handlers/start.js      ← обработчик /start (существующий, расширить)
backend/src/modules/bot/       ← новый модуль для bot-специфичной логики
  routes.js                    ← POST /bot/onboarding/complete
  controller.js
  service.js                   ← onboardingService.complete(telegramId)
```

**Алгоритм:**

1. `/start` → проверка `users.onboarding_completed_at`
2. NULL → запустить сцену онбординга
3. NOT NULL → отправить стандартное меню
4. При нажатии «Открыть приложение» → MiniApp вызывает `POST /api/public/bot/onboarding/complete`
5. Сервис обновляет `users.onboarding_completed_at = NOW()`

**Альтернатива (проще):** При первом открытии MiniApp, если `onboarding_completed_at IS NULL` — MiniApp показывает онбординг-оверлей и при закрытии вызывает API. Бот при этом просто открывает MiniApp.

### 5.7 Edge Cases

| Ситуация                                                      | Поведение                                                                                              |
| ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| Код приглашения просрочен/использован                         | «Эта ссылка уже неактивна. Попросите менеджера создать новую.»                                         |
| Пользователь нажал «Назад» в браузере во время онбординга     | Онбординг в MiniApp не завершён, `onboarding_completed_at` не записан → следующий вход снова онбординг |
| Пользователь уже есть в системе (повторный /start)            | Пропустить онбординг, показать меню бота                                                               |
| Пользователь нажал /start invite_CODE, но уже зарегистрирован | Показать «Вы уже в системе» + стандартное меню                                                         |
| Бот перезапущен, сцена потеряна                               | Stateless запрос к БД — проверяем `onboarding_completed_at`                                            |

## 6. Уведомление о новом курсе

### 6.1 Цель функции

Автоматически уведомить пользователей о публикации нового курса, которого они ещё не проходили. Увеличить охват новых курсов в первые дни после публикации.

### 6.2 Триггер отправки

**Событие:** Администратор переводит курс из статуса `draft` / `archived` в `published`.

**Дополнительный триггер:** Назначение конкретного курса конкретному пользователю/группе (если будет реализована логика `course_assignments`).

**Когда НЕ отправлять:**

- Курс обновлён (изменено описание, добавлена тема), но не опубликован заново.
- Пользователь уже начал или завершил этот курс.
- Пользователь отключил уведомления (`notifications_enabled = false`).
- Пользователь заблокировал бота.

### 6.3 Получатели

**MVP:** Все пользователи с ролью `employee`, у которых:

- `onboarding_completed_at IS NOT NULL`
- `notifications_enabled = true`
- Нет записи в `course_user_progress` для данного курса

**Расширение:** Фильтрация по должности (`position_id`) и филиалу (`branch_id`) если курс назначен определённым группам.

### 6.4 Текст уведомления

```
📚 Новый курс доступен!

«[Название курса]»

[Краткое описание курса — первые 100 символов]

⏱ Примерное время: [X] минут
📝 Тест в конце курса

[Начать курс] ← кнопка WebApp с deep link на конкретный курс
```

**С дедлайном:**

```
📚 Новый курс доступен!

«[Название курса]»

[Описание]

⏰ Пройди до: [дата дедлайна]

[Начать курс]
```

### 6.5 Бизнес-правила

1. Одно уведомление на одного пользователя на один курс (дедупликация).
2. Уведомление отправляется не в момент публикации, а с задержкой 5–15 минут (batching, чтобы не перегружать Telegram API).
3. Если курс архивирован до отправки уведомления — уведомление не отправляется.
4. Время отправки: с 9:00 до 20:00 по таймзоне пользователя (`users.timezone`). Если таймзона не указана — по московскому времени (UTC+3).
5. Ограничение: не более 3 уведомлений о новых курсах в день на одного пользователя.

### 6.6 Техническая реализация

```
backend/src/services/notifications/
  notificationService.js    ← ядро: очередь, отправка, логирование
  courseNotification.js     ← логика уведомлений о курсах
  notificationQueue.js      ← очередь (in-memory или Redis, см. ниже)

backend/src/modules/courses/
  (расширить) service.js    ← при publishCourse() → emit('course.published', courseId)

backend/database/migrations/
  027_add_bot_notifications.sql
  028_add_notifications_settings_to_users.sql
```

**Поток:**

1. Admin нажимает «Опубликовать» → `courses.status = 'published'`
2. `courseService.publish()` добавляет задачу в `bot_notifications` (status: `pending`)
3. Cron-задача каждые 5 минут:
   - Выбирает `pending` уведомления
   - Проверяет часовой пояс пользователя (рабочие часы)
   - Отправляет через Telegram Bot API
   - Обновляет статус: `sent` / `failed`

**Deep link в кнопке:**

```
https://t.me/theorica_bot/app?startapp=course_[courseId]
```

MiniApp при открытии с `startapp=course_123` открывает карточку курса напрямую.

### 6.7 Edge Cases

| Ситуация                                     | Поведение                                                                               |
| -------------------------------------------- | --------------------------------------------------------------------------------------- |
| Пользователь заблокировал бота               | Telegram API вернёт `403 Forbidden` → статус `blocked`, `notifications_enabled = false` |
| Курс удалён до отправки                      | Cron проверяет `courses.status` перед отправкой → пропускает                            |
| Пользователь уже начал курс (race condition) | Проверка `course_user_progress` перед отправкой                                         |
| Массовая публикация 10 курсов за раз         | Throttling: не более 20 сообщений/секунду (лимит Telegram), batching по очереди         |
| Таймзона не задана                           | Fallback: UTC+3 (Москва)                                                                |
| Уведомление не доставлено (сетевая ошибка)   | Retry: 3 попытки с backoff 5/15/60 минут, затем статус `failed`                         |

## 7. Напоминание о прохождении курса

### 7.1 Цель функции

Напомнить пользователю о курсе, который он:

- Получил уведомление, но не начал (0% прогресса)
- Начал, но не завершил (1–99% прогресса)

Цель — повысить completion rate и снизить «забытые» курсы.

### 7.2 Когда отправлять напоминания

**Сценарий A: Курс не начат**

- Через 3 дня после уведомления о курсе, если прогресс = 0%
- Через 7 дней, если прогресс всё ещё = 0% (последнее напоминание)

**Сценарий B: Курс начат, но не завершён**

- Через 2 дня после последней активности в курсе
- За 3 дня до дедлайна (если `courses.close_at` задан)
- За 1 день до дедлайна (срочное напоминание)

**Сценарий C: Дедлайн истёк**

- НЕ отправлять напоминания после `close_at`
- Отправить уведомление о пропущенном дедлайне (одно, сразу)

### 7.3 Логика дедлайнов

```
Если courses.close_at IS NOT NULL:
  - За 7 дней: первое напоминание (если не начат)
  - За 3 дня: второе напоминание (если < 50% прогресса)
  - За 1 день: финальное напоминание (urgent)
  - После close_at: уведомление о пропущенном дедлайне

Если courses.close_at IS NULL (открытый курс):
  - Напоминание через N дней неактивности (настраивается в system_settings)
  - Максимум 3 напоминания на курс на пользователя
```

### 7.4 Тексты уведомлений

**Первое напоминание (не начат):**

```
👋 Привет, [Имя]!

Ты ещё не начал курс «[Название]».

Это займёт всего [X] минут. Начни сейчас — не откладывай обучение!

[Открыть курс]
```

**Напоминание (в процессе):**

```
📖 [Имя], ты на полпути!

Курс «[Название]»: [X]% пройдено.

Осталось совсем немного. Продолжи с того места, где остановился.

[Продолжить курс]
```

**За 3 дня до дедлайна:**

```
⏰ До дедлайна осталось 3 дня!

Курс «[Название]» нужно завершить до [дата].

Прогресс: [X]%

Не теряй время — пройди курс сейчас.

[Открыть курс]
```

**За 1 день до дедлайна:**

```
🚨 Последний шанс!

Завтра истекает срок прохождения курса «[Название]».

Успей пройти сегодня!

[Открыть курс]
```

**После дедлайна (пропущен):**

```
😔 Дедлайн по курсу «[Название]» истёк.

Курс был доступен до [дата]. Обратись к своему менеджеру, если нужен доступ.
```

### 7.5 Частота уведомлений

- Не более **1 напоминания в день** на пользователя на курс.
- Не более **3 напоминаний всего** по одному курсу (если нет дедлайна).
- Если дедлайн есть — можно отправить до 3 напоминаний по расписанию (7 дней, 3 дня, 1 день).
- «Тихий режим»: уведомления только с 9:00 до 20:00 по таймзоне пользователя.

### 7.6 Техническая реализация

```sql
-- Cron: каждые 4 часа
SELECT
  u.telegram_id,
  c.id AS course_id,
  c.title,
  c.close_at,
  COALESCE(cup.progress_percent, 0) AS progress,
  cup.last_activity_at
FROM users u
JOIN courses c ON c.status = 'published'
LEFT JOIN course_user_progress cup ON cup.user_id = u.id AND cup.course_id = c.id
LEFT JOIN bot_notifications bn ON bn.user_id = u.id
  AND bn.entity_type = 'course'
  AND bn.entity_id = c.id
  AND bn.type = 'reminder'
  AND bn.created_at > NOW() - INTERVAL 1 DAY
WHERE
  u.notifications_enabled = true
  AND (cup.completed_at IS NULL OR cup.progress_percent < 100)
  AND (c.close_at IS NULL OR c.close_at > NOW())
  AND bn.id IS NULL  -- ещё не отправляли сегодня
```

**Планировщик:** Node.js `node-cron` или отдельный `scheduler.js` сервис.

### 7.7 Edge Cases

| Ситуация                                           | Поведение                                          |
| -------------------------------------------------- | -------------------------------------------------- |
| Пользователь завершил курс между check и отправкой | Проверка прогресса непосредственно перед отправкой |
| Курс снят с публикации                             | Cron фильтрует только `status = 'published'`       |
| Пользователь отписался от уведомлений              | `notifications_enabled = false` → пропуск          |
| Дедлайн сдвинут администратором                    | Пересчёт расписания напоминаний при следующем cron |
| Несколько курсов с дедлайном в один день           | Отправлять по одному с интервалом 30 минут         |

## 8. Уведомление о результатах курса

### 8.1 Цель функции

Немедленно сообщить пользователю о результате финального теста курса: прошёл или не прошёл. Дать следующий шаг: получить сертификат (при успехе) или пересдать (при провале).

### 8.2 Когда отправляется результат

**Триггер:** Завершение попытки аттестации (`assessment_attempts.status = 'completed'`), если эта аттестация является `final_assessment_id` какого-либо курса.

Уведомление отправляется **немедленно** (не через cron), через event-driven подход.

### 8.3 Варианты результата

| Результат                 | Условие                                                         | Следующий шаг                     |
| ------------------------- | --------------------------------------------------------------- | --------------------------------- |
| Успех                     | `score_percent >= pass_score_percent`                           | Предложить получить сертификат    |
| Провал                    | `score_percent < pass_score_percent` И `remaining_attempts > 0` | Предложить пересдать              |
| Провал, попытки кончились | `score_percent < pass_score_percent` И `remaining_attempts = 0` | Связаться с менеджером            |
| Превышен лимит времени    | `status = 'timeout'`                                            | Результат засчитан, сообщить итог |

### 8.4 Тексты сообщений

**Успех:**


### 8.5 Административная логика

- Менеджер получает сводку о результатах своих сотрудников (не в реальном времени — ежедневный дайджест или в Admin Panel).
- Администратор может сбросить попытки пользователю вручную через Admin Panel.
- В Admin Panel отображается `assessment_attempts` с фильтром по курсу, пользователю, статусу.

### 8.6 Техническая реализация

```
backend/src/modules/assessment/
  (расширить) service.js  ← при completeAttempt() → emit('attempt.completed', data)

backend/src/events/
  assessmentEvents.js     ← подписка на attempt.completed
    → если это финальный тест курса → отправить уведомление
    → если score >= pass → создать сертификат (draft)

backend/src/services/notifications/
  resultNotification.js   ← формирование и отправка уведомления о результате
```

**Важно:** Уведомление отправляется синхронно через event emitter, но сама отправка в Telegram — асинхронно (не блокирует ответ API).

### 8.7 Edge Cases

| Ситуация                                               | Поведение                                                                 |
| ------------------------------------------------------ | ------------------------------------------------------------------------- |
| Пользователь заблокировал бота                         | Записать `bot_notifications.status = 'blocked'`, не отправлять            |
| Пользователь прошёл курс повторно (уже был сертификат) | Проверить `certificates`: обновить дату, не создавать дубликат            |
| Аттестация не является финальным тестом курса          | Уведомление в бот НЕ отправляется (промежуточный тест — только в MiniApp) |
| Сбой при отправке                                      | Записать в `bot_notifications` со статусом `failed` → retry через 5 минут |

## 9. Выдача сертификата

### 9.1 Цель функции

Выдать пользователю официальное подтверждение прохождения курса после успешной сдачи финального теста. Сертификат — ценностная единица: его можно скачать, сохранить, показать.

### 9.2 Условия выдачи

1. Пользователь завершил финальную аттестацию курса.
2. Результат `score_percent >= assessments.pass_score_percent`.
3. Курс имеет флаг `certificate_enabled = true` (новое поле).
4. Сертификат для данного пользователя и курса ещё не выдан (или выдан ранее — перевыпустить с новой датой, если прошёл повторно).

### 9.3 Формат сертификата

**MVP — PDF-файл:**

- Шаблон: HTML → PDF (библиотека `puppeteer` или `pdf-lib`).
- Данные на сертификате:
  - Логотип компании
  - ФИО сотрудника
  - Название курса
  - Дата прохождения
  - Уникальный номер сертификата (UUID)
  - QR-код для верификации (ссылка на публичный endpoint)
- Файл сохраняется в `backend/uploads/certificates/` или AWS S3 (существующая интеграция).

**Расширение:**

- Кастомный шаблон на курс (загружается администратором).
- Электронная подпись / печать.

### 9.4 Пользовательский сценарий получения

```
1. Пользователь получает уведомление с кнопкой [Получить сертификат]

2. Нажимает → переходит в MiniApp на страницу результатов

3. В MiniApp: кнопка [Скачать сертификат]
   → GET /api/public/certificates/:uuid/download
   → Telegram.WebApp.downloadFile() или открыть PDF в браузере

4. Альтернатива в боте:
   Пользователь пишет /certificate → бот отправляет PDF-файл последнего курса
   Или: inline кнопка «Мои сертификаты» → список курсов с сертификатами
```

**Команды бота:**

- `/certificate` — получить последний сертификат
- `/certificates` — список всех сертификатов пользователя

### 9.5 Административное управление

- В Admin Panel: список выданных сертификатов с фильтрами (курс, пользователь, дата).
- Администратор может аннулировать сертификат (`status = 'revoked'`).
- Администратор может перевыпустить сертификат вручную.
- Настройка шаблона сертификата через Admin Panel (загрузка HTML-шаблона).
- Флаг `certificate_enabled` на курсе — включить/выключить сертификаты.

### 9.6 Техническая реализация

```
backend/src/modules/certificates/   ← новый модуль
  index.js
  routes.js
    GET  /api/public/certificates/:uuid          ← метаданные
    GET  /api/public/certificates/:uuid/download ← PDF файл
    GET  /api/public/certificates/my             ← список моих сертификатов
    POST /api/admin/certificates/:id/revoke      ← аннулировать
    POST /api/admin/certificates/issue           ← выпустить вручную
  controller.js
  service.js
    generateCertificate(userId, courseId, attemptId)
    revokeCertificate(certificateId)
    getCertificateByUUID(uuid)
  repository.js
  generator/
    pdfGenerator.js      ← puppeteer/pdf-lib
    templateRenderer.js  ← Handlebars/Mustache для HTML-шаблона

backend/uploads/certificates/        ← локальное хранение (MVP)
```

**Поток генерации:**

1. `assessmentEvents.js` ловит `attempt.completed` с успехом
2. Вызывает `certificateService.generateCertificate(userId, courseId, attemptId)`
3. Генерируется UUID, заполняется шаблон, рендерится PDF
4. PDF сохраняется в файловую систему / S3
5. Запись в таблицу `certificates`
6. `notificationService` отправляет уведомление с кнопкой

**QR-код:** содержит URL вида `https://api.example.com/verify/CERT-UUID`  
Публичный endpoint без авторизации возвращает статус сертификата (валиден/аннулирован + ФИО + курс).

### 9.7 Edge Cases

| Ситуация                                           | Поведение                                                                         |
| -------------------------------------------------- | --------------------------------------------------------------------------------- |
| Пользователь прошёл курс повторно                  | Обновить `certificates.issued_at`, перегенерировать PDF, старый файл архивировать |
| Сертификат уже существует                          | `INSERT ... ON DUPLICATE KEY UPDATE` — не создавать дубликат                      |
| Ошибка генерации PDF                               | Записать `certificates.status = 'generation_failed'`, retry через 10 минут        |
| Пользователь запрашивает сертификат до генерации   | Статус `pending` → «Сертификат готовится, попробуй через минуту»                  |
| Курс изменён после выдачи сертификата              | Сертификат не пересчитывается — зафиксированы данные на момент выдачи             |
| QR-код проверяется для аннулированного сертификата | Возвращать `{"valid": false, "reason": "revoked"}`                                |
| S3 недоступен                                      | Fallback: сохранить в локальную файловую систему                                  |
| Шаблон не настроен                                 | Использовать дефолтный шаблон из `backend/templates/certificate.default.html`     |

```sql
CREATE TABLE bot_notifications (
  id              BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,

                  ) NOT NULL,

  attempt_count   TINYINT UNSIGNED NOT NULL DEFAULT 0,

```

#### `certificates` — выданные сертификаты

```sql
CREATE TABLE certificates (
  id              INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  uuid            CHAR(36) NOT NULL UNIQUE,    -- публичный идентификатор для QR
  user_id         INT UNSIGNED NOT NULL,
  course_id       INT UNSIGNED NOT NULL,
  attempt_id      INT UNSIGNED NOT NULL,       -- за какую попытку выдан
  status          ENUM('pending', 'issued', 'revoked', 'generation_failed')
                    NOT NULL DEFAULT 'pending',
  issued_at       DATETIME NULL,
  revoked_at      DATETIME NULL,
  revoked_by      INT UNSIGNED NULL,           -- admin user_id
  file_path       VARCHAR(500) NULL,           -- путь к PDF
  file_url        VARCHAR(500) NULL,           -- публичный URL
  score_percent   DECIMAL(5,2) NOT NULL,       -- зафиксированный результат
  snapshot_data   JSON NOT NULL,               -- ФИО, курс, дата — на момент выдачи
  created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  FOREIGN KEY (course_id) REFERENCES courses(id),
  FOREIGN KEY (attempt_id) REFERENCES assessment_attempts(id),
  UNIQUE KEY uq_user_course (user_id, course_id),  -- один актуальный сертификат

);
```

## 13. Очереди, cron-задачи и планировщик уведомлений

### Архитектура (MVP — без Redis)

```
backend/src/
  services/
    notifications/
      notificationService.js    ← ядро: CRUD по bot_notifications
      notificationSender.js     ← отправка через Telegram Bot API
      notificationScheduler.js  ← cron-задачи
    certificates/
      certificateService.js
      pdfGenerator.js
  events/
    assessmentEvents.js         ← слушатель событий
    courseEvents.js
```

### Cron-задачи

| Задача                        | Расписание          | Действие                                              |
| ----------------------------- | ------------------- | ----------------------------------------------------- |
| `processPendingNotifications` | Каждые 5 минут      | Отправить pending уведомления с учётом тихого времени |
| `retryFailedNotifications`    | Каждые 30 минут     | Повторить failed (attempt_count < 3)                  |
| `scheduleReminders`           | Каждые 4 часа       | Создать pending напоминания для незавершённых курсов  |
| `deadlineReminders`           | Каждый день в 08:00 | Напоминания о дедлайнах (за 7/3/1 день)               |
| `retryCertificates`           | Каждые 15 минут     | Повторить generation_failed сертификаты               |

### Throttling Telegram API

- Лимит Telegram: 30 сообщений в секунду (разным пользователям).
- Реализация: задержка `setTimeout(fn, index * 35)` при batch-отправке.
- При ошибке `429 Too Many Requests` — Telegram вернёт `retry_after` → соблюдать его.

## 15. Риски и ограничения

| Риск                                           | Вероятность | Влияние | Митигация                                         |
| ---------------------------------------------- | ----------- | ------- | ------------------------------------------------- |
| Пользователь заблокировал бота                 | Средняя     | Средняя | Отслеживать `blocked`, не слать уведомления       |
| Telegram API rate limit                        | Средняя     | Высокая | Throttling + exponential backoff                  |
| Генерация PDF медленная                        | Низкая      | Средняя | Асинхронная генерация, статус `pending`           |
| Не сохранён `chat_id` для проактивных отправок | Низкая      | Высокая | `telegram_id = chat_id` в личных чатах — стандарт |
| Timezone не задан у пользователя               | Высокая     | Низкая  | Fallback UTC+3, предложить указать при онбординге |
| Дублирование уведомлений при retry             | Средняя     | Средняя | `UNIQUE` индекс + idempotency check               |
| Сертификат с устаревшими данными               | Низкая      | Средняя | `snapshot_data` фиксирует данные на момент выдачи |

## 16. Рекомендации по реализации MVP

**MVP включает:**

1. Онбординг (упрощённый): 1 приветственное сообщение + кнопка MiniApp + запись `onboarding_completed_at`.
2. Уведомление о новом курсе: при публикации курса → создать `bot_notifications` → cron отправляет.
3. Напоминание: 1 напоминание через 3 дня неактивности. Дедлайн: за 1 день.
4. Уведомление о результате: синхронно при завершении попытки → отправить в бот.
5. Сертификат: PDF с дефолтным шаблоном, отправить файлом в бот + кнопка в MiniApp.

**MVP НЕ включает:**

## MVP

**Минимальная ценная версия, готовая к запуску:**

1. **Онбординг** — одно приветственное сообщение с кнопкой MiniApp + запись `onboarding_completed_at`.
2. **Уведомление о новом курсе** — при публикации курса → отправить всем подходящим пользователям.
3. **Напоминание** — одно напоминание через 3 дня неактивности + срочное за 1 день до дедлайна.
4. **Результат** — уведомление сразу после завершения теста (прошёл/не прошёл + баллы).
5. **Сертификат** — PDF с дефолтным шаблоном, отправить файлом в бот.

**Не входит в MVP:** QR-верификация, кастомные шаблоны, тихие часы (кроме default 22:00–09:00), дайджест менеджера.

## Рекомендации

1. **Начни с миграций и `bot_notifications`** — это фундамент для всей системы уведомлений. Без него нельзя гарантировать доставку и отслеживать статусы.

2. **Используй event-driven подход** — уже есть паттерн в проекте. Расширь `assessmentService` через EventEmitter, не нарушая существующую архитектуру.

3. **Не используй Redis в MVP** — `bot_notifications` в MySQL достаточно для начала. Redis добавить при нагрузке > 10k пользователей.

4. **Puppeteer для PDF** — уже есть `cloud-storage` модуль, интеграция с S3 готова. Puppeteer + S3 = надёжное решение для сертификатов.

5. **Тестируй на реальном Telegram аккаунте** перед релизом — Telegraf ошибки доставки нужно проверять вживую (rate limit, blocked, chat_id).

6. **Зафиксируй данные сертификата в `snapshot_data`** — ФИО, название курса, результат на момент выдачи. Это критично: если данные пользователя изменятся, сертификат останется корректным.

7. **Онбординг реализуй последним** — это наименее приоритетная функция. Приоритет: уведомления о результатах → сертификаты → напоминания → уведомления о курсах → онбординг.
