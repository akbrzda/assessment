# Инструкция: унифицированная интеграция MAX и Telegram

> Версия: 1.0  
> Дата: 2026-05-16  
> Статус: к согласованию

## 1. Цель документа

Документ фиксирует целевую модель, в которой MAX и Telegram работают одинаково в ключевых сценариях:

- создание ссылки-приглашения,
- переход по ссылке и подтверждение аккаунта,
- обязательный номер телефона,
- двусторонняя идентификация по телефону,
- открытие единого профиля при входе из любой платформы.

## 2. Бизнес-требование

Система должна позволять сотруднику использовать один профиль независимо от того, с какой платформы он заходит (Telegram или MAX). Идентификация должна опираться на подтвержденный номер телефона.

## 3. Базовые принципы реализации

1. Единый пользовательский профиль в домене системы.
2. Отдельные платформенные идентичности (Telegram и MAX) привязываются к одному профилю.
3. Номер телефона обязателен и должен быть подтвержден платформой.
4. Формат хранения номера: E.164.
5. Текущий рабочий scope: только номера РФ в формате +7XXXXXXXXXX.
6. Если с платформы номер приходит в другом формате, он сначала нормализуется в E.164, затем сравнивается.
7. Данные платформы считаются доверенными только после криптографической проверки, предусмотренной документацией платформы.
8. Инвайт-код един для бизнес-процесса, но ссылки формируются отдельно под каждую платформу.

## 4. Унифицированный поток приглашения (обязательно)

### 4.1 Создание приглашения

При создании приглашения менеджер/администратор обязан указать:

- имя,
- фамилию,
- филиал,
- должность,
- номер телефона.

Номер телефона в приглашении становится обязательным атрибутом и считается эталонным для подтверждения аккаунта.

### 4.2 Переход пользователя по ссылке

Пользователь открывает платформенную ссылку:

- Telegram deep link,
- MAX deep link (`startapp` payload).

В обоих случаях в приложение попадает инвайт-код.

### 4.3 Нажатие «Подтвердить»

После перехода по ссылке пользователь нажимает действие подтверждения приглашения.

Система обязательно запрашивает номер телефона через платформенный request-механизм:

- Telegram: контактный сценарий платформы,
- MAX: request_contact/requestContact в соответствии с документацией MAX.

### 4.4 Проверка номера

На сервере выполняется двусторонняя проверка:

1. Проверка подлинности платформенных данных (подпись initData/контактного payload).
2. Нормализация номера из платформы в E.164 (в текущем scope только +7).
3. Сравнение нормализованного номера с номером, сохраненным в приглашении.

### 4.5 Результат проверки

- При совпадении номера:
  - аккаунт подтверждается,
  - pending-профиль активируется,
  - платформенная идентичность привязывается к единому профилю,
  - пользователь получает доступ к профилю.

- При несовпадении номера:
  - активация запрещается,
  - пользователю показывается понятная ошибка,
  - попытка фиксируется в аудит-логе.

## 5. Двусторонняя идентификация между MAX и Telegram

### 5.1 Сценарий Telegram -> MAX

Если пользователь уже зарегистрирован через Telegram и открывает MAX:

1. MAX-идентичность проходит валидацию.
2. Запрашивается и подтверждается номер телефона.
3. Номер сравнивается с номером существующего профиля.
4. При совпадении MAX-идентичность привязывается к существующему профилю.

### 5.2 Сценарий MAX -> Telegram

Если пользователь уже зарегистрирован через MAX и открывает Telegram:

1. Telegram-идентичность проходит валидацию.
2. Запрашивается и подтверждается номер телефона.
3. Номер сравнивается с номером существующего профиля.
4. При совпадении Telegram-идентичность привязывается к существующему профилю.

## 6. Обязательные проверки безопасности

1. Серверная валидация подписи initData для каждой платформы.
2. Серверная проверка подлинности контактных данных для каждой платформы.
3. Нормализация номера телефона в едином формате перед сравнением.
4. Защита от повторов и race condition при активации pending-профиля.
5. Лимиты попыток подтверждения в текущем scope не применяются.
6. Логирование событий подтверждения и отказа без утечки чувствительных данных.

## 7. Плюсы решения

1. Один сотрудник = один профиль в системе.
2. Снижение дублей пользователей между платформами.
3. Предсказуемый процесс активации по инвайту.
4. Повышенная надежность идентификации за счет проверки номера на стороне платформы.

## 8. Минусы и сложности

1. Рост сложности auth-слоя и миграций данных.
2. Необходимость обработки конфликтов исторических данных по телефонам.
3. Дополнительные UX-состояния при ошибках подтверждения номера.
4. Более строгие требования к аудиту и антифроду.

## 9. Рекомендации к внедрению

1. Внедрять поэтапно через режим совместимости со старым потоком.
2. Сначала добавить модель multi-platform identity, затем включать строгую проверку номера.
3. Выполнить единообразную нормализацию исторических номеров к формату E.164 (+7) перед включением строгой сверки.
4. Закрепить единые тексты ошибок и статусы подтверждения для обеих платформ.

## 10. Критерии приемки

1. Приглашение не создается без номера телефона.
2. Активация по инвайту невозможна без request-подтверждения номера.
3. При совпадении номера аккаунт активируется и открывается профиль.
4. При несовпадении номера показывается короткая и понятная ошибка, активация не выполняется.
5. Пользователь, зарегистрированный в Telegram, открывает тот же профиль в MAX после подтверждения номера.
6. Пользователь, зарегистрированный в MAX, открывает тот же профиль в Telegram после подтверждения номера.

## 11. Область изменений

- Backend: auth, invitations, identity-linking, audit, миграции.
- Frontend MiniApp: платформа-агностичный auth bootstrap, UX подтверждения номера.
- Bot-слой: parity сценариев кнопок и приглашений для Telegram и MAX.
- Документация и QA: единые тест-кейсы для двух платформ.

## 13. Контракт подтверждения номера (Telegram + MAX)

### 13.1 Входные данные (обязательные)

1. `platform`: `telegram` или `max`.
2. `inviteCode`: строка инвайта из deep link/startapp payload.
3. `initData`: платформенный initData payload для серверной проверки подписи.
4. `contactPayload`: подтвержденные платформой данные контакта (request_contact/requestContact).

### 13.2 Проверки на backend (строгий порядок)

1. Проверить подпись `initData` по правилам платформы.
2. Проверить подлинность `contactPayload` по правилам платформы.
3. Извлечь номер и нормализовать в E.164.
4. Для текущего scope принять только формат `+7XXXXXXXXXX`.
5. Сравнить с телефоном в приглашении.

### 13.3 Результат

1. При успехе:
   - активировать pending-профиль;
   - привязать платформенную идентичность;
   - выставить статус подтверждения телефона `verified`;
   - вернуть успешный ответ с единым профилем.
2. При ошибке:
   - не активировать профиль;
   - вернуть короткую ошибку несоответствия;
   - записать событие в аудит.

## 14. Критерии линковки существующего профиля по телефону

1. Линковка разрешена только для подтвержденного телефона из платформенного request-механизма.
2. Номер после нормализации должен точно совпадать с `users.phone_e164`.
3. Если найден ровно один активный профиль:
   - текущая платформенная идентичность привязывается к нему;
   - новый профиль не создается.
4. Если найдено несколько профилей с одним номером:
   - auto-link запрещен;
   - возвращается конфликт;
   - создается запись в аудит для ручной диагностики.
5. Если профиль не найден:
   - применяется invite-flow (для новых пользователей) без авто-линковки к произвольной записи.

## 15. План миграции и отката

### 15.1 Миграция

1. Применить SQL-миграцию schema-first:
   - `users.phone_e164`, `users.phone_verification_status`, `users.phone_verified_at`, `users.phone_verification_source`;
   - таблицу `user_platform_identities` с уникальными ограничениями.
2. Выполнить backfill Telegram-идентичностей из `users.telegram_id`.
3. Добавить индексы на `users.phone_e164` и `invitations.phone`.
4. Включить запись новых идентичностей в auth-потоке.
5. После стабилизации включить строгую сверку телефона на подтверждении.

### 15.2 Откат

1. Отключить запись в identity-слой feature-флагом/конфигом приложения.
2. Откатить release на предыдущую версию backend.
3. Данные в `user_platform_identities` не удалять автоматически; использовать как диагностический слой.
4. При критичном инциденте выполнить ручной rollback миграции только после бэкапа БД.
5. После отката повторно прогнать smoke и авторизационные проверки для Telegram-потока.
