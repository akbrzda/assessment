# Roadmap: интеграция MAX + Telegram с единым профилем

> Версия: 1.0  
> Дата: 2026-05-16  
> Горизонт: 6 спринтов (по 1 неделе)  
> Статус: план

## Sprint 0: Уточнение и фиксация контрактов

### Цель

Зафиксировать финальные правила идентификации и данные для миграции.

### Задачи

- [x] Подтвердить целевой формат номера телефона (E.164, текущий scope: только РФ +7).
- [x] Подтвердить UX для ошибки несовпадения номера (короткий и понятный текст).
- [x] Зафиксировать контракт подтверждения номера для Telegram и MAX.
- [x] Зафиксировать критерии линковки существующего профиля по телефону.
- [x] Подготовить план миграции и отката.

### Выход спринта

Подписанный архитектурный и продуктовый baseline по интеграции.

## Sprint 1: Подготовка данных и модель идентичностей

### Цель

Подготовить backend к multi-platform идентификации.

### Задачи

- [x] Добавить/обновить сущности для хранения платформенных идентичностей.
- [x] Ввести обязательность телефона в приглашениях.
- [x] Добавить статусы подтверждения телефона и источники подтверждения.
- [x] Подготовить миграции для существующих пользователей.
- [x] Добавить индексы и ограничения целостности.

### Выход спринта

Схема БД готова к одновременной работе MAX и Telegram.

## Sprint 2: Унифицированный invite-flow и проверка номера

### Цель

Включить новый обязательный сценарий подтверждения номера по инвайту.

### Задачи

- [x] Обновить backend-валидации: инвайт без телефона запрещен.
- [x] Реализовать серверный процесс request-подтверждения номера.
- [x] Реализовать нормализацию номера из request в E.164 (+7) и сверку с номером в приглашении.
- [x] Реализовать активацию pending-профиля только при совпадении номера.
- [x] Реализовать стандартную ошибку при несовпадении номера.
- [x] Добавить аудит-логи для успешной/неуспешной верификации.

### Выход спринта

Новый инвайт-поток работает на backend с обязательным подтверждением номера.

## Sprint 3: MAX MiniApp и бот-паритет

### Цель

Подключить MAX как полноценный канал с поведением, идентичным Telegram.

### Задачи

- [x] Реализовать bootstrap MAX MiniApp с серверной валидацией initData.
- [x] Подключить обработку deep link `startapp` с invite payload.
- [x] Подключить request_contact/requestContact для MAX.
- [x] Реализовать формирование ссылок приглашения для MAX.
- [x] Привести тексты и состояния UX к паритету с Telegram.

### Выход спринта

MAX-поток приглашения и подтверждения аккаунта работает end-to-end.

## Sprint 4: Кросс-платформенная линковка профилей

### Цель

Гарантировать открытие одного и того же профиля из обеих платформ.

### Задачи

- [x] Реализовать логику auto-link по подтвержденному номеру.
- [x] Реализовать безопасный fallback для конфликтов.
- [x] Добавить защиту от дублирующей активации и гонок.
- [x] Добавить админ-инструменты диагностики конфликтов идентичности.
- [x] Прогнать миграцию/дедупликацию существующих данных.

### Выход спринта

Профиль корректно открывается из MAX и Telegram по одному номеру телефона.

## Sprint 5: QA, hardening и запуск

### Цель

Закрыть риски и выпустить функциональность в production.

### Задачи

- [x] Провести интеграционные тесты Telegram <-> MAX сценариев.
- [x] Провести негативные тесты (несовпадение номера, различный входной формат номера, устаревшие ссылки).
- [x] Проверить rate limiting и антифрод ограничения.
- [x] Проверить полноту и корректность аудит-логов.
- [x] Подготовить операционный runbook для поддержки.
- [x] Подготовить релиз-чеклист и план мониторинга.

### Выход спринта

Функциональность стабилизирована и готова к выпуску.

## Ключевые риски

1. Конфликты старых данных по телефонам.
2. Несогласованность UX в Telegram и MAX.
3. Ошибки в криптографической валидации контактных данных.
4. Дубли пользователей при гонках активации.

## Критерии готовности релиза

1. Все тест-кейсы линковки профиля прошли в Telegram и MAX.
2. Ошибки несовпадения номера отрабатывают корректно и предсказуемо.
3. Сверка номера после нормализации в E.164 (+7) работает одинаково для MAX и Telegram.
4. Операционная команда получила runbook и сценарии диагностики.
